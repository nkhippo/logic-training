## 1. アプリの目的

ビジネス現場での意思決定・問題解決において、不完全な情報の中から仮説を立て、
論理を連鎖させて判断・行動を導く力を鍛えるWebアプリ。

書籍では得られないAIによるインタラクティブな問題生成と個別フィードバックで、
以下の3つの論理力を体系的に磨く。

- **読む力**：相手の文章・発言の論理的な構造を把握し、飛躍や穴を見抜く
- **判断する力**：不完全・曖昧な情報から仮説を連鎖させ、妥当な解釈と行動を導く
- **伝える力**：論理的に筋の通ったメッセージを、読み手に合わせた形で表現する

## 2. このアプリが生まれた背景

### 2-1. 開発者の課題感

開発者（Naoya）はビジネス現場でロジカルシンキングに課題を感じ、書籍によって学ぼうとした。
しかし書籍では以下の課題が解消できないと判断し、このアプリを開発した。

ビジネス現場でのロジカルシンキングの課題は、主に以下2つの方向で現れた。

**インプット側（読む・聞く）**
- 相手の論理の飛躍に気づけず「なんとなく理解」してしまい、後で手戻りになる
- 相手が述べた前提・根拠を精査せずに受け入れてしまう

**アウトプット側（書く・話す）**
- 報告書のAgendaを作るとき、文章と文章をつなぎ合わせられず、ストーリーの「筋」が通せない
- 自分が論理の飛躍した議論を展開してしまい、指摘を受けることがある

### 2-2. 書籍の限界

| 書籍の限界 | 具体的な問題 |
|---|---|
| 題材のズレ | 書籍の問題は論文・評論・レポートが中心。ビジネス現場でロジカルシンキングが問われるのはチャット・メール・口頭での報連相（報告・連絡・相談）であり、レポートを読む・書く機会は相対的に少ない |
| 問題形式のズレ | 書籍はリアルタイムフィードバックが不可能なため、情報量が少なく・確度が高い問題に集中しがち。結果として単発の正解が定まりやすい問題に終始する |
| 実務との乖離 | ビジネス現場では情報が不完全・曖昧な状態で多くの情報を処理し、論理を複数積み上げてアウトプットを出さなければならない。単発の回答でアウトプットが完結する機会はほぼない |
| 個別最適化の欠如 | 問題数・難易度・フィードバックが固定されており、自分の課題に合わせられない |

### 2-3. このアプリの解決策

| 書籍の限界 | このアプリの解決策 |
|---|---|
| 題材がビジネス現場と乖離 | ビジネス文書（メール・報告書・提案書・議事録など）を題材にした問題を生成 |
| 情報が少なく単発問題に終始 | 不完全な情報の中で仮説を連鎖させる多段階問題・往復形式の問題（積み上げタブ）を提供 |
| 問題数・難易度が固定 | 難易度1〜5・AIが毎回異なる問題を生成するため、問題数は無限に近い |
| 個別フィードバックがない | AIが回答者の状況・回答内容に合わせた採点・アドバイスをリアルタイムで提供 |

### ターゲットから除外する像

- 「正しい日本語・文章術を身につけたい人」は対象外とする。このアプリの目的は「論理の構造を鍛えること」であり、文章の正確さ・美しさを目的にする人とはコンセプトが異なる。混在するとアプリのコンセプトがブレるため、明示的に除外する。

## 4. 磨きたい論理力の定義

このアプリが鍛える「論理力」を以下の3軸で定義する。
問題設計・採点基準・フィードバックの方針は、常にこの3軸に基づくこと。

### 4-1. 読む力（A）

相手の文章・発言の論理的な構造を把握し、飛躍や穴を見抜く力。

- 主張・根拠・前提の関係を正確に把握する
- 「AだからB」という推論の飛躍・前提の欠如を発見する
- 接続詞・段落構成から文章の論理的な流れを読み取る

### 4-2. 判断する力（D）

不完全・曖昧な情報から仮説を連鎖させ、妥当な解釈と行動を導く力。

これはこのアプリが最も重視する力であり、書籍では鍛えにくい部分でもある。

**「仮説の連鎖」とは何か：**
ビジネス現場では情報が常に不完全・曖昧であり、「確実な答え」が最初から存在することは少ない。そのとき必要なのは、以下のようなプロセスだ。

```
事実・情報（不完全・曖昧）
  ↓ 何が確かかを整理する（事実の仕分け）
一次仮説（おそらくこうだ）
  ↓ 仮説から新たな解釈を導く
二次仮説（ならばこうなるはずだ）
  ↓ さらに積み上げる
三次仮説→判断・アウトプット
```

このプロセスを単発で終わらせず、複数段階を連鎖させることが「積み上げ」であり、
ビジネス現場で実際に求められる推論の形だ。

**想定する主な場面（優先順）：**
1. 意思決定の場面：「A案とB案のどちらを選ぶべきか」を判断する
2. 問題解決の場面：「売上が落ちた原因は何か」を分析し、打ち手を考える

### 4-3. 伝える力（B）

論理的に筋の通ったメッセージを、読み手に合わせた形で表現する力。

- 主張→根拠→結論の流れで文章を構成する
- 読み手が「次に何をすべきか」を先回りして盛り込む
- 情報を目的に沿って整理・構造化し、読みやすい形で表現する

## 7. 機能要件

### 7-1. 共通機能

| 機能 | 内容 |
|---|---|
| 問題生成 | Claude APIを呼び出しAIが問題文・設問を生成 |
| 採点・フィードバック | Claude APIで回答を採点し解説を表示。設問別評価・飛躍指摘・別解・改善例・総合講評まで**途中で切れない**よう、返却トークン上限を設問数・難易度に応じて確保する（2026-05時点で従来比約1.5倍） |
| 過去問保存 | GAS経由でGoogle Spreadsheetに保存 |
| 過去問一覧・再挑戦 | 保存済み問題を一覧表示し、再度解答可能 |
| 削除 | 過去問の削除（GAS経由） |
| ガイドモーダル | 各タブの使い方をMarkdownから表示（日本語: `guide/*.md`、英語: `guide/en/*.md`） |
| 言語切替 | 日本語・英語の切替（i18n） |
| テーマ（必須） | 全タブでUI表記「テーマ」に統一。穴埋め・要約・批判読み＝文書タイプ、空雨傘＝テーマ領域、積み上げ＝場面タイプのプリセット。5列レイアウト。初期未選択・押下時バリデーション |
| 業界（任意） | 全タブで文書タイプと難易度の間に配置。`st.industry` を共有。生成時に問題オブジェクトへ保存し、**問題セクション**にテーマ・業界・難易度を表示 |
| 難易度選択 | 5段階（入門〜超難問）。初期は未選択。説明欄は問題方針のみ表示（穴抜き数・設問数などの自動表示はUIに出さない） |
| 答え合わせ | 回答欄が空の場合はAPIを呼ばず入力を促す |
| 要約のUI順序 | 文書タイプ → 業界 → 難易度 → ボリューム（難易度4以上のみ） |
| 生成成功後 | 作成条件（文書タイプ・業界・難易度・ボリューム・積み上げ場面）を初期化 |
| API処理中 | 全画面オーバーレイで操作不可（タブ・過去問・ガイド含む）。スクロールのみ可能 |
| 印刷 | 問題・解説の印刷機能 |
| ペルソナ設定 | ヘッダーの設定ボタン→モーダルで設定。業界×職種（MAX3つ・プリセット選択）と勤続年数（数値選択）を設定できる。全タブ横断で共通。問題のテーマ選択（業界）とは独立した概念。localStorageに保存。問題生成時にペルソナ情報をプロンプトに付加し、回答者のバックグラウンドに沿った題材・表現で問題を生成する。難易度はペルソナによって変わらない。 |

### 7-2. タブ別の固有機能

#### 穴埋め
- 全難易度：400〜500字固定
- 穴抜き数：難易度1=2個、2=3個、3=3個、4=4個、5=5個
- 難易度1〜2：ヒントあり
- 採点後：完全補完文・日常例文・代替接続詞の論旨変化を表示

#### 要約
- 設問タイプ：語句説明・主張要約・理由説明の3種
- 設問数：難易度1=1問、2=2問、3〜5=3問
- 文字数制限チェック：オーバー時は提出不可
- 回答方法：テキスト入力 or 写真提出（最大2枚）

#### 批判読み
- 難易度1〜2：B形式（短い論証3問・独立）
- 難易度3〜5：A形式（論説文1本から4〜5問）
- 設問タイプ：飛躍の指摘・条件の補完・反論への応答・論証構造の整理・対偶の検証
- 各設問にツールチップ（概念説明）あり
- **ビジネスモード**：対偶の検証は出題しない。難易度4〜5に「立場が異なる人からの疑問」を追加

#### 積み上げ

**概要：**
不完全・曖昧な情報（定量データ＋人の発言・状況説明の混在）を与え、
仮説を連鎖させて意思決定・問題解決の判断を導くタブ。
AIは「最終アウトプットの品質向上に向けて部下を導く上司」の立場で対話する（答えは教えない）。

**題材の形式：**
- デフォルト：定量データ＋人の発言・状況説明の混在（形式C）
- 将来的に形式A（データ・数値のみ）・形式B（発言・定性情報のみ）も選択可能にする

**STEP構成：**

| STEP | 内容 | 適用難易度 |
|---|---|---|
| STEP1-A | 確かな事実・不確かな事実・矛盾の整理 | 全難易度 |
| STEP1-B | 現時点の判断とその理由（最終でない場合は判断のバリエーション） | 全難易度 |
| STEP1-C | 矛盾解消のために誰に・何を確認するか | 難易度3以上・追加情報あり問題のみ |
| STEP以降（上司の問い返し） | 前STEPの不備に対してAIが問い返す（4パターン） | 不備がある場合のみ |
| 追加情報 | 仮説を揺さぶる新情報を提示・再度A・B問 | 難易度3以上 |
| 最終STEP-一言要約 | 判断を20〜30字で表現（具体と抽象の往復） | 難易度2以上 |
| 最終STEP-最終判断 | 論理モード（結論＋根拠）または伝達モード（バイネームメッセージ） | 全難易度 |

**上司の問い返しパターン（4種）：**

| パターン | 発動条件 | AIの問い |
|---|---|---|
| 具体が足りない | 抽象的な表現のみで具体的状況が示されていない | 「具体的にはどういう状況ですか？」 |
| 抽象化が足りない（So What?） | データを羅列しているだけで解釈に至っていない | 「それはつまり何を意味しますか？」 |
| 論理の飛躍 | AからCへの間のステップが飛ばされている | 「〇〇の場合はどうなりますか？」 |
| 反証の見落とし | 一方向の仮説のみで反証可能性を考えていない | 「〇〇という可能性はないですか？」（難易度4以上のみ） |

**難易度設計：**

| 難易度 | STEP数 | 追加情報 | STEP1-C | 反証問い返し | 一言要約 |
|---|:---:|:---:|:---:|:---:|:---:|
| 1（入門） | 1 | なし | なし | なし | なし |
| 2（基礎） | 1〜2 | なし | なし | なし | あり（最終） |
| 3（標準） | 2 | あり（一次判断後） | あり | なし | あり（最終） |
| 4（上級） | 2〜3 | あり（二次判断後） | あり | あり | あり（最終） |
| 5（超難問） | 3 | あり（複数回） | あり | あり | あり（最終） |

**採点の5軸：**
- 事実と解釈の仕分けの正確さ
- 仮説の論理的妥当性（飛躍がないか）
- 追加情報を受けて仮説を適切に更新できたか（難易度3以上）
- 最終判断の根拠の充実度
- 一言要約の抽象化の精度（難易度2以上）

**最終判断の回答モード（ユーザーが選択）：**
- 論理モード：結論と根拠を端的に記述。伝え方は評価しない
- 伝達モード：バイネームで送ることを想定したメッセージ形式。伝え方も評価対象

**気配りタブとの関係：**
気配りタブは積み上げタブの「伝達モード」として統合した。
伝え方のトレーニングはこのアプリの主軸（論理力）に対するサブテーマとして位置づけ、
回答者がオプションとして選択できる形にする。

#### 空雨傘
- 全難易度：300〜400字固定
- 難易度1〜3：帰納型、難易度4〜5：演繹型混在
- 設問タイプ：空（事実の仕分け）・雨（解釈）・傘（行動）・導出の説明・自己検証・法則の限界
- **事実の仕分け（難易度3以上）**：回答UIを「事実として書かれている部分」「解釈・意見として書かれている部分」の**2つのテキスト欄**に分割（難易度2は事実のみの記事のため1欄のまま）
- **「仮説の連鎖」との関係**：空雨傘は1段階の推論（事実→解釈→行動）を正確に行う訓練。将来的には多段階の仮説連鎖を問う発展問題の追加を検討する

## 8. AI問題設計の方針

このセクションはCursorおよびClaudeへの指針として明示する。

### 8-1. ビジネス題材の原則

- 全タブの問題文・設問文はビジネス現場で実際に起こりうる状況を題材にする
- 評論文・哲学的文章・論文調の題材は**ビジネスモードでは使用しない**（受験モードで使用）
- 業界を指定した場合でも、**その業界の専門知識・専門用語・規制・法律を知らなくても読める内容**にする。業界は「文脈の色付け」のみに使用する

### 8-2. 設問文・フィードバックの言葉遣い

- 設問文・フィードバックはビジネス現場で自然に使われる言葉に限定する
- 論理学の専門用語（演繹・帰納・対偶・論証など）は設問文・フィードバック文には使わない
- 「何をすればよいか」が直感的にわかる表現にする
- ツールチップ（❓）は専門用語の説明ではなく、設問の取り組み方のヒントを示す

### 8-3. 温度設定（temperature）

- 問題生成：0.9（多様な問題が生成されるよう創造性を確保）
- 採点・フィードバック：0.3（採点基準が一貫するよう低めに設定）

### 8-4. 「仮説の連鎖」を問う問題設計の指針

判断する力（D）を鍛える問題（批判読み・空雨傘・積み上げ）は、以下を意識して設計する。

- 情報を1つ与えて1つの答えを導く「単発問題」にしない
- 複数の情報・条件を組み合わせて初めて妥当な判断が導けるよう設計する
- 同じ情報から複数の解釈が成立しうる場合はその可能性を採点フィードバックで示す
- 難易度が上がるにつれて「情報の曖昧さ・複雑さ」を高め、推論のステップ数を増やす

### 8-5. 積み上げタブの問題設計指針

- 題材は「どうすべきか・どう判断できるか」を問う設計にする
- 追加情報が入る問題は、最初の情報をより曖昧・不完全にして判断を絞り込めないようにする
- AIは採点者ではなく「上司」として対話する。正解を教えず、問い返しによって回答者自身が気づくよう促す
- 難易度が上がるにつれて情報の曖昧さ・複雑さを高め、仮説のステップ数を増やす
- ペルソナが設定されている場合は、そのバックグラウンドに沿った題材・事例を使用する
- ペルソナはあくまで題材の「文脈の色付け」。難易度・採点基準はペルソナによって変わらない

### 8-6. ペルソナ設定の問題設計への反映方針

ペルソナが設定されている場合、問題生成プロンプトに以下を付加する。

```
回答者のバックグラウンド：[業界×職種]（勤続[N]年）
このバックグラウンドに沿った題材・事例・表現レベルで問題を生成すること。
ただし難易度・採点基準はバックグラウンドによって変えないこと。
```

勤続年数からスキルレベルを類推する基準：
- 1年未満：基本的な業界用語は知っているが実務経験は浅い
- 1〜3年：日常業務の流れは把握しているが専門的判断は経験不足
- 3〜5年：専門知識を持ち、自律的に業務を遂行できる
- 5〜10年：チームや案件をリードできる経験を持つ
- 10年以上：マネジメント・意思決定の経験が豊富
