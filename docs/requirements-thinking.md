## 1. サービスの目的

断片的で曖昧な状況の中から「何を問うべきか」を自分で定義し、
仮説を積み重ねながら判断・行動を導く力を鍛えるWebアプリ。

AIとの対話を通じて思考プロセスを繰り返し積み上げることで、
仕事の現場で使える判断力と、問いを立てる力を育てることを目的とする。

## 2. なぜこのサービスが必要か

### 2-1. 仕事における根本的な難しさ

仕事で本当に難しいのは、答えを出すことではなく、
**何を問うべきかを決めること**にある。

どれだけ論理的に正確な答えを出しても、解くべき問いが間違っていれば、
その答えに意味はない。コンサルタントであれば「問いの設定を誤る」ことの
コストを肌で知っているはずである。クライアントへの最終プレゼンで、
そもそも解くべき問いが違っていたと気づいたときの後戻りは大きい。
このサービスは、その経験への処方箋として設計されている。

実際のビジネスでは、問いが明確に与えられる場面はほとんどない。

- 状況は断片的で、情報はいつも不完全
- ステークホルダーの利害は複雑に絡み合う
- 「何を解くべきか」という問い自体、誰も定義してくれない

そのような状況でも、判断を下し、次の行動を決めなければならない。
その判断の質を最終的に決めるのは、問いを立てる精度である。

### 2-2. 書籍でロジカルシンキングを学んでも実践で活かせない理由

書籍を読んでロジカルシンキングの型は理解できた。
しかし、いざ仕事で使おうとすると上手くいかない——
そういった経験をしたことがある人は多いはずである。

その理由は以下の3点に集約される。

**問いが最初から用意されている**
書籍の問題は「この文章の論点は何か」「なぜAはBと言えるか」という形式をとる。
問いが最初から与えられている問題を解き続けても、
自分で問いを立てる力は育たない。

**情報が整理された単発問題しかない**
書籍はリアルタイムのフィードバックができないため、
情報が整理されていて正解が1つに定まる問題に集中する。
曖昧な情報を前に仮説を積み重ねていくプロセスは、書籍では訓練できない。

**自分の思考への個別フィードバックがない**
「どこで論理が飛んだか」「何を見落としたか」を
その場で指摘してもらえる環境が、書籍には存在しない。

### 2-3. AIとの対話形式が必要な理由

書籍の限界を超えるためには、AIとの対話形式が不可欠である。

**問いが与えられない問題を作れる**
AIは「断片的な状況だけを提示し、何を解くべきかをユーザーが定義する」
問題を生成できる。こうした問題は、書籍では構造的に作れない。

**ユーザーの回答に応じて問いを動的に生成できる**
「一次判断→新情報の追加→判断の修正→最終判断」という
段階的な思考プロセスを訓練するには、
ユーザーの回答を受けてリアルタイムで反応できる対話相手が必要である。
書籍の解説は一方通行であり、この役割を果たせない。

**見落としをその場で指摘し、その代償を示せる**
AIはユーザーの回答を読んで、どの思考ステップが抜けているかを検出し、
「その判断、3ヶ月後に覆る可能性はないか」という形で
見落としの影響を具体的に示すことができる。
この体験が、なぜその思考ステップが必要かという気づきにつながる。

**回答者に合わせたフィードバックを提供できる**
同じ問題でも、回答者の思考の傾向に応じた指摘ができる。
書籍の解説は全員に同じ内容を届けるが、
AIは「あなたがこう考えたから、この部分が弱くなっている」という
個別の指摘を提供できる。

### 2-4. AIが「採点者」ではなく「上司」として振る舞う理由

AIの立場には2つの選択肢がある。

- **採点者**：回答に正解・不正解の判定を下し、模範解答を示す
- **上司**：答えを教えず、論理の弱点に対して問い返しを行い、本人に気づかせる

このサービスでは後者を採用する。
理由は、ビジネス現場で人が成長するのは
「正解を教わったとき」ではなく
「上司から問い返しを受けて自分で気づいたとき」だからである。
特にコンサル業界では、上司との対話によって思考が鍛えられる経験が
スキル形成の中核を成している。

AIが採点者として振る舞うと、ユーザーは「正解を覚える」学習になりやすい。
上司として振る舞うことで、ユーザーは「自分の思考の癖に気づき修正する」
学習へと自然に誘導される。

## 4. 思考力の定義と設計思想

### 4-1. このトレーニングが最終的に目指すこと

このトレーニングの最終的なゴールは、
**自分で状況を読み、「何を解くべきか」という問いを立てられるようになること**である。

「判断力を鍛える」ことを中心に置きながら、
その先にある「問いを自分で立てる」というスキルまでを射程に収めている。

仕事の現場で「問いが与えられる」ことはほとんどない。
「なんとなく営業チームの勢いがない」「ここ数ヶ月、クライアントの反応が鈍い」——
実際に与えられるのは、こうした曖昧な状況だけである。
そこから「今、自分は何を解くべきか」を定義し、
必要な情報を集め、判断を積み重ねていく。
この一連のプロセスを鍛えることが、このサービスの設計方針である。

### 4-2. なぜ6つの思考タイプが必要か

最終的なゴールは「問いを立てられるようになること」だが、
問いを立てる力は判断のプロセスを知らないと育たない。
何を解くべきかを決めるためには、人がどのように判断を進めているのかを
構造として理解している必要がある。
そのため、まず判断のプロセスを6つの操作として整理した。

実際の仕事で判断や問題解決を行うとき、
意識しているかどうかに関わらず、思考の操作は以下の6つに集約される。
これをこのトレーニングでは「6つの思考タイプ」と呼ぶ。

| タイプ | 何をするか |
|---|---|
| ①状況を整理する | 今何が起きているかを、事実・不確かな情報・矛盾に分けて読む |
| ②原因を掘り下げる | なぜそうなったかを、表面の原因から根本まで掘り下げる |
| ③先を読む | このまま進むとどうなるか、何が起きうるかを見積もる |
| ④選択肢を比べる | 何が最善かを、自分なりの基準を持って判断する |
| ⑤相手を動かす | 誰がどう動くかを想定し、伝え方・順序を設計する |
| ⑥問いを疑う | そもそも今解こうとしている問いが正しいかを立ち止まって確認する |

この6タイプを設計した理由は「設問の再現性」にある。
どんな業界・場面でも同じ問いかけの型が使えるため、
問題を生成するAIへの指示が安定し、採点の基準もブレにくくなる。

また、この6タイプは独立して存在するものではなく、
判断のプロセスの中で以下のように連なる。

```
①状況を整理する（今何が起きているかを正確に把握する）
　↓
②原因を掘り下げる、または③先を読む、または⑤相手を動かす
（状況に応じて、縦に深めるか・横に広げるかを判断する）
　↓
④選択肢を比べる（判断を下す）
　↓
⑥問いを疑う（そもそもこの問いでよかったかを確認する）
```

この流れが、すべての問題に共通する骨格となる。

### 4-3. なぜ4つのレベルが必要か

6つの思考タイプを「知っている」だけでは、仕事で使えるようにはならない。
使えるようになるまでには、以下の4段階を経る必要がある。

**レベル1：型を実際に使ってみる**
設問が与えられた状態で解く。6タイプの「何を考えるか」が指定されており、
それぞれの型の使い方を体感する段階。

**レベル2：型の中で何を確認するかを自分で決める**
使うタイプは指定されるが、「何を整理するか」は自分で決める。
「①状況を整理する」と言われたとき、事実・矛盾・関係者の立場など、
何を確認すべきかを自分で問いかける訓練。

**レベル3：どの型をどの順序で使うかを自分で選ぶ**
状況だけが与えられ、どのタイプをどの順序で使うかを自分で判断する。
同じ状況でも「どの型で解くか」を誤ると、判断の方向が狂う。
型の選択判断そのものがスキルになる段階。

**レベル4：何を解くべきかを自分で定義する**
断片的で曖昧な状況だけが提示される。
「何を解くべきか」という問いを定義することから始まる。
これが最終的なゴールである。

レベル1〜4は、段階的に「問いを与えられる量を減らしていく」設計である。

### 4-4. 問いを立てる力は繰り返すだけでは身につかない

繰り返し練習することで身につくのは、「型を正確に実行する精度」にとどまる。

たとえば「①状況を整理する」という設問に繰り返し答えることで、
事実・不確かな情報・矛盾を区別する操作の精度は上がる。
しかし「この場面では①から始めるべきか、⑥から始めるべきか」という
型の選択や、「そもそも何を解くべきか」という問いの設定は、
繰り返しだけでは育たない。

問いを立てるには、「なぜその情報が必要なのか」という目的意識が伴っている必要がある。

この目的意識を育むには、以下の2つのプロセスが欠かせない。

**①「型を省いた結果どうなるか」を体験すること**
型を省いた状態で判断を下したとき、
「その選択肢、3ヶ月後に覆る可能性はないか」という問いかけを受けることで、
省いた型の意味を自分で気づくきっかけが生まれる。
このトレーニングでは、AIが「上司」の立場から
答えを教えずに問い返すことで、この体験を提供する。

**②最終フィードバックで、省いたことの影響を具体的に示すこと**
「②原因を掘り下げるを省くと、直接原因への対処しか行われず、
同じ問題が繰り返す構造が残る」という形で、
省いた型の具体的な影響を言語化して示す。
これが「なぜその型が必要か」の理解につながり、
次の問題に取り組むときの目的意識として機能する。

## 5. 問題設計の全体構造

### 5-1. 問いの核心（10タイプ）

問いの核心は「どんなビジネス場面の問いを解くか」を定義する補助概念である。
問題を作る際にユーザーが任意で選択でき（デフォルトはランダム）、
題材文の生成精度と採点基準の一貫性を高めるために使用する。
ユーザーには見せず、内部処理にのみ使う。

問いの核心は1つの問題につき1つだけ選択する。
複数同時選択は許容しない（題材生成の方向性が定まらなくなるため）。

| 問いの核心 | 問いの本質 | 主に使う思考タイプ |
|---|---|---|
| 実現可否型 | これはできるか？ | ④＋①③ |
| 優先順位型 | どれを先にやるか？ | ④＋① |
| 合意形成型 | どうすれば通るか？ | ⑤＋①④ |
| 原因分析＋ゴール定義型 | なぜ起きたか・何を目指すか？ | ②＋①④ |
| リスク見積もり型 | 何が起きうるか？ | ③＋①④ |
| 価値の定義型 | それは本当に良いことか？ | ⑥＋①② |
| 構造の発見型 | なぜ繰り返すのか？ | ②＋① |
| 視点の転換型 | 別の立場から見るとどう見える？ | ⑤＋⑥ |
| トレードオフの明示型 | 何を諦めるか？ | ④＋③⑥ |
| 前提の問い直し型 | 当たり前は本当に正しいか？ | ⑥＋① |

### 5-2. 題材文の形式

デフォルトは以下の混在形式を採用する。

```
数値・実績などの定量的な情報
＋
会議での発言・メール・ヒアリング内容などの定性的な情報
```

レベルが上がるほど情報は断片的・曖昧になる設計とする。
レベル4では「最近、チームの動きが悪い気がする」という
ほぼ何も与えられない状況から始まる問題を生成する。

### 5-3. 難易度とレベルの対応

| レベル | 何が与えられるか | 自分で決めること | 適用できる難易度 |
|---|---|---|---|
| レベル1 | 題材文＋設問A・B・C | 特になし | 難易度1〜5 |
| レベル2 | 題材文＋使うタイプと順序 | 各タイプで何を確認するか | 難易度3〜5 |
| レベル3 | 題材文＋6タイプの一覧 | どのタイプをどの順序で使うか | 難易度3〜5 |
| レベル4 | 断片的・曖昧な題材文のみ | 問いを立てることから始めるすべて | 難易度3〜5 |

難易度1・2はレベル1固定とする。
難易度3以上ではレベル1〜4をユーザーが自由に選択できる。

### 5-4. 回答の流れと点数ルール

各ステップで回答するたびに、AIが内部で100点満点のスコアをつける。
スコアの数値は表示せず、内部処理にのみ使用する。

```
95点以上　　　　：次のステップへ進む（フィードバックなし）
80点以上95点未満：次のステップへ進む
　　　　　　　　　（1点だけ指摘を受け、直すか次へ進むかを選べる）
80点未満・1回目 ：AIから問い返しを受ける
80点未満・2回目 ：そこで終了
　　　　　　　　　（次のステップには進まず、ここまでのフィードバックを受け取る）
```

レベルごとの詳細なステップ数は仕様書（specification-thinking.md）に記載する。

### 5-5. 最後の問い「つまり、どういうこと？」

難易度2以上のすべてのレベルで、最後に共通して入る問いかけ。

```
目的：
　問題を通じて自分が定義した「解くべき問い」（4-1で示した最終ゴール）に対して、
　「なぜその結論に至ったか・次に何をすべきか」を
　端的に言語化できるかを確認する。

形式：
　AIがユーザーの問いをもとに質問文を動的に生成する。
　例：「この依頼を3週間以内に実現できるかどうかを判断する」
　　　という問いに対して、
　　　「この依頼を3週間以内に実現できるかどうか、
　　　なぜそう判断したか・次に何をすべきかを
　　　2文・100字以内でまとめてください」と生成する。

点数ルール（90点ルール）：
　90点以上　　 ：最終フィードバックへ
　90点未満・1回目：1回だけ問い返しを受ける
　90点未満・2回目：最終フィードバックへ（終了ではなく通常のフィードバック）
```

**難易度1で除外する理由：**
難易度1は「型の使い方を体感する」入門段階であり、
ここまでの分析を圧縮して言語化する負荷は学習目的に対して重すぎる。
そのため、最後の問いは難易度2以上でのみ提示する。

## 6. 採点の設計

### 6-1. AIの立場

採点者ではなく、「上司」として動作する（2-4で述べた背景に基づく）。

```
目的　：最終的な判断の質を上げるために、部下の思考を引き出す
制約　：答えを直接教えない。「ここが悪い」と一方的に批判しない
やり方：省いた型の影響を具体的に示す問いかけで、
　　　　自分で気づかせることを優先する
```

### 6-2. 採点の役割構成

```
役割A：点数の判定（内部処理のみ）
　├ レベル別の評価基準を適用する
　├ 問いの核心に照らして回答の方向を確認する
　├ 論理的な整合性をチェックする（後述）
　├ ユーザーが定義した問い（user_core）を抽出する（レベル2〜4）
　└ JSONで返す：点数 / 合否 / 省いた思考タイプ / 論理の問題 / 判定理由 / user_core

役割B：問い返しの生成
　├ 95点以上：次へ進む（問い返しなし）
　├ 80〜95点：指摘1つ＋「直す」か「次へ進む」かを選べる
　├ 80点未満：省いた型の影響を具体的に示す問い返しを生成する
　└ 点数の数値はユーザーへのメッセージに含めない

役割C＋D1：最終フィードバック＋振り返りへの問いかけ（1回のAPI呼び出し）
　├ パート1：最終フィードバック
　│　├ できていたこと
　│　├ 省いた視点とその影響（具体的に。たとえば
　│　│　「②原因を掘り下げるを省くと、直接原因への対処しか行われず、
　│　│　 同じ問題が繰り返す構造が残る」という形で示す）
　│　├ 論理の整合性への指摘
　│　├ 別の考え方の提示
　│　└「つまり、どういうこと？」への評価
　└ パート2：振り返りへの問いかけ（ENABLE_REFLECTION=trueのみ）
　　　└ 以下の3パターンから1〜2問を選んで提示する
　　　　 ・パターンX（省略・混同への指摘）：
　　　　 　「[ステップ]で[具体的な問題]がありました。
　　　　 　　そのときの判断の根拠を振り返ってみてください」
　　　　 ・パターンY（自己批判の促し）：
　　　　 　「今回の回答全体で、最も『本当にこれで良いか』と
　　　　 　　迷った場面はどこですか？」
　　　　 ・パターンZ（思い込みの指摘）：
　　　　 　「[ステップ]で[前提]を確認せずに進んだ場面がありました。
　　　　 　　なぜそこを疑わなかったと思いますか？」

役割D2：振り返り1回目のフィードバック＋追加の問いかけ（任意・点数なし）
　├ ユーザーの回答から「質問があるか/ないか」をAIが読み取る
　└ 必ず問いかけで締める（D1とは異なるパターンを選ぶ）

役割D3：振り返り2回目のフィードバック（任意・点数なし・問い返しなし）
　├ ユーザーの回答から「質問があるか/ないか」をAIが読み取る
　└ この問題全体を通じた気づきを1つ提示して締める
```

### 6-3. 論理的な整合性のチェック

以下の3つの層でチェック項目を定義し、`js/thinking/domain.js` の
`LOGIC_CHECK_MAP` に保持する。
採点のたびに、問いの核心と使った思考タイプに合わせて動的に組み合わせて使う。

**全問題に共通するチェック：**

- 事実と自分の解釈・意見が混ざっていないか
- 結論と根拠の間に飛躍がないか（AだからBと言えるか）
- 原因と結果の向きが逆になっていないか

**思考タイプ別のチェック（例）：**

- ①状況の整理：事実・不確かな情報・矛盾の3つに分けられているか
- ②原因の掘り下げ：表面の原因で止まらず、根本まで掘り下げられているか
- ④選択肢の比較：評価の基準が明示されているか、何を諦めるかが意識されているか
- ⑤相手を動かす：自分の論理を押しつけていないか、相手が動く条件から考えているか

**問いの核心別のチェック（例）：**

- 実現可否型：「できる/できない」の判断基準が明示されているか
- 合意形成型：主な関係者全員が考慮されているか
- 原因分析型：どこまで直すかが意識的に選ばれているか

### 6-4. ユーザーが定義した問い（user_core）の扱い

最終問い「つまり、どういうこと？」の質問文をAIが動的に生成する際、
ユーザーが定義した問いを起点とする必要がある。
この問いを `user_core` と呼び、以下の方針で抽出・保持する。

| レベル | user_coreの取得方法 |
|---|---|
| レベル1 | システム側のデフォルト（問いの核心ごとに定義された定型文）を使用 |
| レベル2 | 切り口定義ステップの採点APIで抽出 |
| レベル3 | タイプ選択ステップの採点APIで抽出 |
| レベル4 | 問い定義ステップの採点APIで抽出 |

採点APIのJSONレスポンスに `userCore` フィールドを含め、
そこから取得する。
0回目（または-1回目）で打ち切りになった場合は、
システム側のデフォルトにフォールバックする。

## 7. 振り返り機能

### 7-1. 目的

「省いた型の影響を知る」にとどまらず、
「なぜ自分はそこを省いたのか」を言語化することを促す。

自分の思考の傾向に気づくには、思考プロセスを後から振り返り
「なぜそう考えたか」を言葉にする経験を積む必要がある。
この機能はその場を提供するために設計されている。
4-4で述べた「目的意識を育む」プロセスを、
最終フィードバック後の対話として実現する位置づけにある。

### 7-2. フロー

```
1. 最終回答を送信する
2. AIから最終フィードバック（役割C）＋振り返りへの問いかけ（役割D1）が届く
3. 問いかけに回答する（任意）
　 → 回答しない場合は「スキップして終了」を選び、ここでセッションを終える
4. AIからフィードバック＋さらに深める問いかけ（役割D2）が届く
5. 問いかけに回答する（任意）
　 → 回答しない場合はここでセッションを終える
6. AIから最後のフィードバック（役割D3）が届く（問い返しなし・示唆で締める）
```

ユーザーが任意のタイミングで離脱できる設計とし、
回答を強制しない。離脱時は4・5でセッションを正常終了させる。

### 7-3. 制御フラグ

```js
// js/shared/01-config.js
const ENABLE_REFLECTION = true; // falseで振り返り機能を全体的に無効化
```

振り返り機能はセッションとして重くなるため、
状況に応じて公開・停止を切り替えられる設計とする。
初回は `true`（公開）で運用する。

## 8. 機能要件

### 8-1. 共通機能

| 機能 | 内容 |
|---|---|
| 問題生成 | Claude APIで題材文・設問文をその都度生成（設問文も固定テンプレートを使わない） |
| 採点・フィードバック | 役割A〜D3を分けて処理する |
| 過去問の保存 | GAS経由でGoogle Spreadsheetに保存 |
| 言語切替 | ヘッダーの「言語」ボタン→モーダルで日本語・英語を選択（i18n対応） |
| ペルソナ設定 | ヘッダーの「プロフィール」ボタン→モーダル。業界×職種（最大3つ）・勤続年数。論理トレーニングと共有（localStorage） |
| 業界選択 | 全問題で共通・任意。問題の文脈に反映する |
| 振り返り機能 | ENABLE_REFLECTIONフラグで制御 |

### 8-2. ペルソナと業界の使い分け

ペルソナと業界は別の概念として扱う。

| 概念 | 範囲 | 目的 | 設定UI |
|---|---|---|---|
| ペルソナ | 全タブ横断・全アプリ共通 | 回答者の背景に沿った表現で題材を生成 | ヘッダーのプロフィールボタン |
| 業界（テーマ） | 問題ごと | この問題の題材の文脈の色付け | 問題作成エリア |

**ペルソナ：**
「あなたはどんな人ですか？」を定義する。
業界×職種・勤続年数で構成される。
1度設定すれば全タブ・全問題に共通で適用される。
論理トレーニングと共有するため、localStorageに保存する。

**業界（テーマ）：**
「この問題はどんな業界の話か？」を定義する。
問題ごとに任意で選択する。
専門知識がなくても読める範囲で、文脈の色付けに使う。

両者は独立して機能し、混在を避けるためUI上も別の場所に配置する。

### 8-3. 問題を作るときの設定

```
必須：
　難易度（1〜5）
　レベル（1〜4）　※難易度1〜2はレベル1固定

任意：
　問いの核心（10タイプから選択・デフォルトはランダム）
　　→ ユーザーには表示しない
　　→ 題材文の生成と採点基準の精度向上に使用
　　→ 過去問に紐づけて保存する
　業界（テーマ）
　ペルソナ（設定済みの場合は自動で反映）
```

### 8-4. 過去問の保存項目

```
保存するもの：
　└ 題材文
　└ 設問（レベル1の場合は設問A・B・C、レベル2〜4は問いの構造）
　└ 問いの核心（ユーザーには見せない・内部保持）
　└ user_core（システムデフォルトまたは過去のユーザー定義）
　└ 難易度・レベル
　└ 業界・ペルソナのスナップショット（設定されている場合）

保存しないもの：
　└ ユーザーの回答
　└ 採点フィードバック
　└ 振り返りの内容
```

過去問を保存する主な目的はAPI利用コストの削減である。
他のユーザーが採点結果を参照して答え合わせに使えないよう、
回答とフィードバックは保存しない設計とする。

## 10. 問題設計の方針（AIへの指示原則）

### 10-1. 題材文

- 実際の仕事で起こりうる状況を題材にする
- 「今何を解くべきか」が題材文から直接読み取れないようにする
  （状況の事実と情報だけを記述し、問いは読み手が定義する）
- 業界・ペルソナが設定されている場合は文脈の色付けに使うにとどめる
  （その業界の専門知識がなくても読める内容にする）
- レベルが上がるほど、情報を断片的・曖昧にする

### 10-2. 設問文

設問文はAIがその都度生成する設計とする。
固定テンプレートを使わない理由は以下である。

- 題材ごとに具体的な人物・数値・発言を組み込んだ設問のほうが、
  ユーザーが「何を答えればよいか」を直感的に把握できる
- 固定テンプレートでは題材との一貫性が崩れ、誤回答を誘発しやすい

設問文の生成方針：
- 題材の登場人物・数値・発言を使った具体的な問いかけにする
- 論理学の専門用語（演繹・帰納・対偶など）は使わない
- 「何をすればよいか」が読んでわかる表現にする

### 10-3. 温度設定（temperature）

| 用途 | 設定値 | 理由 |
|---|---|---|
| 問題の生成 | 0.9 | 毎回異なる問題が生成されるよう、幅を持たせる |
| 採点・フィードバック | 0.3 | 採点の基準が一貫するよう、ブレを抑える |
| 振り返りへの問いかけ（役割D1〜D3） | 0.7 | 自然な対話として機能するよう、適度な幅を持たせる |

## 12. 論理トレーニングとの関係

論理トレーニング（logic.html）で身につく力は以下である。

- 文章の接続・論理の流れを正確に読む力（穴埋め・批判読み）
- 主張と根拠を残して情報を圧縮する力（要約）
- 事実から1段階の解釈と行動を導く力（空雨傘）

これらは「与えられた情報を正確に処理する」能力であり、
インプットの精度を高めるトレーニングである。

一方、このサービスが鍛えるのは「処理した情報から判断を導き、行動を決める」
能力であり、インプットからアウトプットへの変換が中心となる。

**前者は「読む」、後者は「考えて決める」**——
この違いが、2つのサービスを別ページとして分離した根拠である。

```
論理トレーニング（logic.html）
　└ 読む力・具体と抽象を行き来する力の基礎を鍛える
　└ 1問5〜10分で完結するトレーニング
　└「思考トレーニングへ →」のリンクを設置

思考トレーニング（thinking.html）
　└ 問いを立てる力・判断する力を鍛える
　└ 1セッション15〜60分の深いトレーニング
　└「論理トレーニングへ ←」のリンクを設置

共有するもの：
　└ ペルソナ設定（localStorage）
　└ 言語設定（localStorage）
　└ GASのスプレッドシート（別シートで管理）
　└ 共通JS（js/shared/）
```

論理トレーニングは思考トレーニングの前提として推奨されるが、
強制はしない。ガイドモーダルから論理トレーニングへの導線を提供し、
ユーザーが必要に応じて行き来できるようにする。
