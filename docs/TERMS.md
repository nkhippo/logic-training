# thinkgrindai — 用語定義

**最終更新**: 2026-05-25  
**目的**: Claude・Cursor・Naoya間で齟齬が生じないよう、本プロジェクトで使う用語を厳密に定義する。

> このファイルに記載のない用語が会話で使われた場合、Claudeは必ず確認すること。  
> Naoyaが曖昧な表現を使った場合も、Claudeは確認質問を行うこと。

---

## 1. ドキュメントの種類

このセクションは特に重要。「要件定義書」「仕様書」と言われた際にどのファイルを指すかを厳密に定義する。

### 1-1. 要件定義書（Requirements）

**ファイル位置**: `docs/requirements/`配下

| ファイル | 何が書かれているか |
|---|---|
| `docs/requirements/common.md` | 両サービス共通の目的・背景・対象ユーザー・AI設計方針・ペルソナ方針 |
| `docs/requirements/logic/overview.md` | 論理トレーニング全体の論理力定義・タブ構成・ロードマップ |
| `docs/requirements/logic/{tab}.md` | 各タブ（fill/summary/critique/ame）の目的・学習体験・難易度別の意図 |
| `docs/requirements/thinking/overview.md` | 思考トレーニングの目的・6タイプ・4レベル・問題設計の思想 |
| `docs/requirements/thinking/scoring.md` | 採点・振り返り設計（役割A〜D3・user_core） |

**書く内容**：
- 「なぜこの機能が必要か」（背景・課題感）
- 「ユーザーが何ができるか」（機能要件）
- 「どんな学習体験を目指すか」（体験設計）
- 「設計上の判断とその理由」

**書かない内容**：
- 具体的な実装値（文字数・APIパラメータ・関数名）
- JSON構造・状態遷移フロー
- データベーススキーマ

### 1-2. 仕様書（Specification）

**ファイル位置**: `docs/specification/`配下

| ファイル | 何が書かれているか |
|---|---|
| `docs/specification/common.md` | 両サービス共通のファイル構成・API共通仕様・ヘッダー・ペルソナUI・GAS共通仕様 |
| `docs/specification/logic/common.md` | 論理トレーニング共通UI（テーマ選択・max_tokens・GASエンドポイント） |
| `docs/specification/logic/{tab}.md` | 各タブの状態遷移・問題生成・採点・プロンプト設計意図・GASカラム |
| `docs/specification/thinking/overview.md` | 状態オブジェクト・問題オブジェクト・画面構成・定数一覧 |
| `docs/specification/thinking/steps.md` | レベル別ステップフロー・80点ルール・90点ルール |
| `docs/specification/thinking/api.md` | 全API呼び出し仕様（パラメータ・プロンプト設計意図） |
| `docs/specification/thinking/data.md` | user_coreフロー・GASカラム・persona_snapshot・残タスク |

**書く内容**：
- 「どのように動くか」（状態遷移・分岐条件）
- API呼び出しのパラメータ（temperature・max_tokens の具体値）
- データ構造（JSON・GASカラム）
- 「なぜそのパラメータか」（プロンプト設計意図）
- UI要素の挙動（バリデーション・エラーハンドリング）

**書かない内容**：
- ユーザーの学習体験・哲学
- コード実装の詳細（JSファイル内コメントへ）
- 開発者の意思決定プロセス（DESIGN_DECISION_HISTORY.md へ）

### 1-3. その他のドキュメント

| ファイル | 役割 | 「要件定義書」「仕様書」と呼ばない理由 |
|---|---|---|
| `docs/PROJECT_CONTEXT.md` | プロジェクトのビジョン・ロードマップ | サービス特定の要件ではなく、プロジェクト全体の方向性 |
| `docs/DEVELOPMENT_POLICY.md` | 開発フロー・タスク分類・運用ルール | 何を作るかではなく、どう作るかの方針 |
| `docs/architecture.md` | フロントエンド構成・ディレクトリ設計 | 実装の構造図であり、機能仕様ではない |
| `docs/gas-column-headers.md` | GASシートのカラム定義（コピペ用） | 仕様書のコピペ補助情報 |
| `docs/adr/*.md` | Architecture Decision Record（設計判断の正本） | 要件定義書・仕様書ではなく、意思決定の記録 |
| `docs/DESIGN_DECISION_HISTORY.md` | 設計判断の経緯メモ | ドキュメントに書かれない「なぜそうなったか」の記録 |
| `docs/_index.md` | ドキュメント索引 | 全ドキュメントの目次 |
| `docs/TERMS.md`（このファイル） | 用語定義 | プロジェクト内の用語集 |
| `docs/DOCUMENT_GUIDELINES.md` | ドキュメント記載粒度のガイドライン | ドキュメント編集時の基準 |
| `docs/cursor-instructions/*.md` | Cursor向け作業指示書 | 実装単位の作業手順 |
| `docs/setup/*.md` | 環境構築ガイド | 開発環境のセットアップ手順 |

## 2. サービス名

| 用語 | 正確な意味 | 同義語・関連語 |
|---|---|---|
| **thinkgrindai** | プロジェクト全体の名称（ドメイン候補名）。論理トレーニング＋思考トレーニングの総称 | サービス全体 |
| **論理トレーニング** | `logic.html` で提供される4タブのトレーニング | logic、Logic Training |
| **思考トレーニング** | `thinking.html` で提供されるトレーニング | thinking、Thinking Training |

「ロジック」「思考力」など類似語が出てきた場合、Claudeはどのサービスを指すか確認すること。

## 3. 機能・概念

### 3-1. タブ

論理トレーニングの構成要素。`logic.html` 内の機能単位。

| タブID | 表示名 | 状態 |
|---|---|---|
| fill | 穴埋め | 実装済み |
| summary | 要約 | 実装済み |
| critique | 批判読み | 実装済み |
| ame | 空雨傘 | 実装済み |

> **削除されたタブ**：
> - kibari（気配り）：2026-05-25に完全削除
> - tsumiaage（積み上げ）：完全削除

### 3-2. レベル・難易度（思考トレーニング）

両者は別概念であり、必ず使い分けること。

| 用語 | 意味 | 範囲 |
|---|---|---|
| **難易度（difficulty）** | 題材文の複雑さ | 1〜5 |
| **レベル（level）** | 問いの自律性（補助輪をどれだけ外すか） | 1〜4 |

両者は直交する2軸として設計されている（詳細は `requirements/thinking/overview.md §3-4`）。

「難しさ」と言われた場合は必ずどちらか確認すること。

### 3-3. 6つの思考タイプ

| ID | タイプ名 |
|---|---|
| 1 | 状況を整理する |
| 2 | 原因を掘り下げる |
| 3 | 先を読む |
| 4 | 選択肢を比べる |
| 5 | 相手を動かす |
| 6 | 問いを疑う |

### 3-4. 問いの核心（10タイプ）

問題設計の補助概念。ユーザーには見せず、内部処理にのみ使う。
詳細は `requirements/thinking/overview.md §4-1` を参照。

### 3-5. user_core

ユーザーが定義した「解くべき問い」を1文で要約したもの。
最終問い「つまり、どういうこと？」の質問文をAIが動的に生成する際の起点として使用する。
詳細は `requirements/thinking/scoring.md §4` を参照。

### 3-6. ペルソナと業界（テーマ）

混同しやすい2概念。必ず使い分けること。

| 概念 | 意味 | 範囲 | 設定UI |
|---|---|---|---|
| **ペルソナ** | 回答者のバックグラウンド（業界×職種×勤続年数） | 全タブ横断・全アプリ共通 | ヘッダーのプロフィールボタン |
| **業界（テーマ）** | この問題の舞台設定 | 問題ごと | 問題作成カード内 |

詳細は `requirements/common.md §7-1`（ペルソナ）を参照。

「業界」と言われた場合、ペルソナの業界（プロフィール設定）か、問題の業界（問題ごとの設定）かを必ず確認すること。

### 3-7. AIの役割（思考トレーニング採点）

| 役割 | 内容 |
|---|---|
| 役割A | 点数の判定（内部処理） |
| 役割B | 問い返しの生成 |
| 役割C | 最終フィードバック |
| 役割D1 | 振り返りへの問いかけ |
| 役割D2 | 振り返り1回目のフィードバック＋追加の問いかけ |
| 役割D3 | 振り返り2回目のフィードバック（示唆で締める） |

詳細は `requirements/thinking/scoring.md §2` を参照。

## 4. 開発プロセス用語

### 4-1. 開発フェーズ

| 用語 | 意味 |
|---|---|
| **Phase 1〜N** | プロジェクト全体の開発フェーズ。`PROJECT_CONTEXT.md` で管理 |
| **フェーズ1〜N** | 個別サービスのロードマップ。各 `requirements/{service}/overview.md` で管理 |

「フェーズ」と言われた場合、プロジェクト全体か個別サービスかを確認すること。

### 4-2. タスク種別

| 用語 | 意味 |
|---|---|
| **タイプA** | 軽微タスク（1日10〜20件） |
| **タイプB** | 中規模案件（検討2〜3日） |
| **タイプC** | 大規模案件（Phase立ち上げ） |

詳細は `DEVELOPMENT_POLICY.md` を参照。

### 4-3. ドキュメント関連

| 用語 | 意味 |
|---|---|
| **要件確定** | Naoyaが要件をGoogle Sheetsの「要件確定シート」に記入した状態 |
| **指示書作成済み** | Claudeが `cursor-instructions/` 配下に作業指示書を作成し、Naoyaが確認した状態 |
| **PR** | Cursorが作成するGitHub Pull Request |

### 4-4. 採点ルール

| 用語 | 意味 | 適用範囲 |
|---|---|---|
| **80点ルール** | 各ステップでの分岐（95/80/<80） | 思考トレーニング全ステップ |
| **90点ルール** | 最終問いの分岐 | 思考トレーニングの最終問い |
| **打ち切り** | 80点未満を2回受けた場合の終了 | 思考トレーニング各ステップ |

詳細は `specification/thinking/steps.md` を参照。

## 6. Claudeへの注意事項

以下の場面でClaudeは**必ず**Naoyaに確認すること。

- 「ドキュメント」と言われたが、要件定義書・仕様書・cursor-instructions・design history のどれを指すか不明な場合
- 「難易度」「レベル」のどちらか不明な場合
- 「業界」がペルソナの業界か問題の業界か不明な場合
- 「フェーズ」がプロジェクト全体か個別サービスか不明な場合
- 「タブ」が現存タブか削除されたタブ（kibari等）を指す可能性がある場合

確認なしに憶測で進めると、ドキュメント間の矛盾を生む。
