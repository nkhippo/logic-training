## 1. 状態遷移フロー

```
[初期状態]
　テーマ・業界・難易度 = 未選択

　↓ テーマ・難易度を選択 → 生成ボタン押下

[問題生成中]
　全画面オーバーレイ

　↓ API応答

[問題表示（A形式：難易度3〜5）]
　メタ行（テーマ・業界・難易度）
　論説文1本
　設問4〜5問（各設問にツールチップあり）
　各設問に：回答欄（テキスト or 写真）

[問題表示（B形式：難易度1〜2）]
　メタ行
　短い論証3本（各論証が独立した設問）

　↓ 全設問に回答 → 答え合わせボタン押下

[採点中] → [採点結果表示]
　設問別フィードバック（飛躍の指摘・別解・改善例）
　総合講評
　過去問保存ボタン
```

## 2. 問題生成仕様

### 2-1. 難易度別パラメータ

| 難易度 | 形式 | 文章量 | 設問数 |
|---|---|---|---|
| 1（入門） | B形式：短い論証3問（独立） | 各論証50〜80字 | 3問 |
| 2（基礎） | B形式：短い論証3問（独立） | 各論証50〜80字 | 3問 |
| 3（標準） | A形式：論説文1本 | 約400字 | 4問 |
| 4（上級） | A形式：論説文1本 | 約400字 | 5問 |
| 5（超難問） | A形式：論説文1本 | 約400字 | 5問 |

> 実装：A形式の文章量は `C_TEXT_LENGTH=400`（`js/logic/04-domain.js`）。B形式の論証は50〜80字（`11-critique.js` 内のプロンプトに固定値）。

### 2-2. 設問タイプ

| タイプ | 出題難易度 | ビジネスモード |
|---|---|---|
| 本当にそう言える？の指摘 | 全難易度 | ✅ |
| 結論が成立するための条件 | 全難易度 | ✅ |
| 反対意見への応答 | 難易度2〜5 | ✅ |
| 話の流れの整理 | 難易度3〜5 | ✅ |
| 立場が異なる人からの疑問 | 難易度4〜5 | ✅（ビジネスのみ） |
| 対偶の検証 | 難易度4〜5 | ❌（受験モードのみ） |

### 2-3. API呼び出し仕様

| 項目 | 値 |
|---|---|
| temperature | 0.9 |
| max_tokens | 2000（問題生成） |

### 2-4. プロンプト設計意図

**B形式とA形式を分ける理由：**  
入門・基礎段階では1つの論証を独立して検証することに集中させる。
A形式（論説文1本から複数問）は文章全体の流れを把握しながら
複数箇所の論理を検証する能力が必要なため、標準以上からに限定する。

**「対偶の検証」をビジネスモードで除外する理由：**  
対偶（「PならばQ」→「QでなければPでない」）の検証は論理学的な操作であり、
ビジネス現場での実践的な論理力とは性格が異なる。
ビジネスモードでは「立場が異なる人からの疑問」という、
よりビジネス文脈に沿った設問に差し替える。

**ツールチップの設計意図：**  
各設問タイプに「❓」ツールチップを付ける。
ツールチップは設問タイプの専門用語説明ではなく、
「この設問で何をすればよいか」の取り組み方のヒントを示す。
