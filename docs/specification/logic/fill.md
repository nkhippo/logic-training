## 1. 状態遷移フロー

```
[初期状態]
　テーマ・業界・難易度 = 未選択

　↓ テーマ・難易度を選択 → 生成ボタンが押下可能に
　↓ 生成ボタン押下（テーマ・難易度未選択の場合は alert）

[問題生成中]
　全画面オーバーレイ（操作不可）

　↓ API応答

[問題表示]
　メタ行（テーマ・業界・難易度）
　問題文（空欄あり）
　各空欄に入力欄
　難易度1〜2のみ：ヒント表示

　↓ 全欄入力 → 答え合わせボタン押下
　↓ 未入力がある場合は入力を促す（API非呼び出し）

[採点中]
　全画面オーバーレイ（操作不可）

　↓ API応答

[採点結果表示]
　各設問の正誤
　完全補完文
　各設問への日常例文
　代替接続詞を使った場合の論旨変化
　過去問保存ボタン
```

## 2. 問題生成仕様

### 2-1. 難易度別パラメータ

| 難易度 | 文章量 | 穴抜き数 | ヒント |
|---|---|---|---|
| 1（入門） | 400〜500字 | 2個 | あり |
| 2（基礎） | 400〜500字 | 3個 | 軽いヒント |
| 3（標準） | 400〜500字 | 3個 | なし |
| 4（上級） | 400〜500字 | 4個 | なし |
| 5（超難問） | 400〜500字 | 5個 | なし |

### 2-2. API呼び出し仕様

| 項目 | 値 |
|---|---|
| temperature | 0.9 |
| max_tokens | 1500（問題生成） |
| systemプロンプトの方針 | 教育目的・ビジネス文書向け・指定文書タイプに沿った題材 |

### 2-3. プロンプト設計意図

**文章量を全難易度で固定する理由：**  
穴埋めは接続詞の選択に集中する訓練であり、文章の長さが学習体験に影響しないようにする。
難易度の差は「穴抜き数」と「文脈の複雑さ」で表現する。

**ヒントを低難易度のみに付与する理由：**  
入門・基礎では「なぜこの接続詞か」の考え方を学ぶため、
ヒントとして接続詞の種類（逆接・因果・要約など）を提示する。
標準以上ではヒントを外し、文脈から推論させる。

## 3. 採点仕様

### 3-1. API呼び出し仕様

| 項目 | 値 |
|---|---|
| temperature | 0.3 |
| max_tokens | 難易度1〜3: 2250 / 難易度4〜5: 3750 |

### 3-2. 採点フィードバックの構成

```
各設問：
　正誤判定
　正解の接続詞
　日常例文（その接続詞の典型的な使用例）
　代替接続詞を使った場合の論旨変化

全体：
　完全補完文（全空欄を正解で埋めた完成版）
```

### 3-3. プロンプト設計意図

**採点temperature=0.3を使う理由：**  
正誤判定は一貫性が重要。creative な応答は不要なため低めに設定。

**代替接続詞の論旨変化を出力する理由：**  
「しかし」を選ぶべき箇所で「つまり」を選んだ場合に文章の意味がどう変わるかを示すことで、
接続詞の選択が論理構造に与える影響をユーザーが体感できる。
正誤だけを教えるより深い理解につながる。

## 4. GASカラム定義（fill シート）

| カラム | 内容 |
|---|---|
| id | タイムスタンプ（Unix ms） |
| theme | テーマ文字列 |
| diff | 難易度（1〜5） |
| date | 保存日時 |
| industry | 業界（任意・未選択時は空） |
| text | 問題文（空欄あり） |
| answers | 正解接続詞リスト（JSON） |
| hints | ヒントリスト（JSON） |
| feedback | 採点フィードバック（JSON） |
| userAnswers | ユーザー回答（JSON） |
| lang | 言語（ja/en） |
