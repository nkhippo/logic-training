## 1. このアプリが生まれた背景

### 1-1. 開発者の課題感

開発者（Naoya）はビジネス現場でロジカルシンキングに課題を感じ、書籍で学ぼうとした。
しかし書籍では以下の課題が解消できないと判断し、このアプリを開発した。

**インプット側（読む・聞く）**
- 相手の論理の飛躍に気づけず「なんとなく理解」してしまい、後で手戻りになる
- 相手が述べた前提・根拠を精査せずに受け入れてしまう

**アウトプット側（書く・話す）**
- 報告書のAgendaを作るとき、文章と文章をつなぎ合わせられず、ストーリーの「筋」が通せない
- 自分が論理の飛躍した議論を展開してしまい、指摘を受けることがある

### 1-2. 書籍の限界

| 書籍の限界 | 具体的な問題 |
|---|---|
| 題材のズレ | 書籍の問題は論文・評論・レポートが中心。ビジネス現場でロジカルシンキングが問われるのはチャット・メール・口頭での報連相であり、レポートを読む・書く機会は相対的に少ない |
| 問題形式のズレ | 書籍はリアルタイムフィードバックが不可能なため、情報量が少なく確度が高い問題に集中しがち。結果として単発の正解が定まりやすい問題に終始する |
| 実務との乖離 | ビジネス現場では情報が不完全・曖昧な状態で多くの情報を処理し、論理を複数積み上げてアウトプットを出さなければならない。単発の回答でアウトプットが完結する機会はほぼない |
| 個別最適化の欠如 | 問題数・難易度・フィードバックが固定されており、自分の課題に合わせられない |

### 1-3. このアプリの解決策

| 書籍の限界 | このアプリの解決策 |
|---|---|
| 題材がビジネス現場と乖離 | ビジネス文書（メール・報告書・提案書・議事録など）を題材にした問題を生成 |
| 情報が少なく単発問題に終始 | 不完全な情報の中で仮説を連鎖させる多段階問題を提供 |
| 問題数・難易度が固定 | 難易度1〜5・AIが毎回異なる問題を生成するため、問題数は無限に近い |
| 個別フィードバックがない | AIが回答者の状況・回答内容に合わせた採点・アドバイスをリアルタイムで提供 |

## 2. 対象ユーザー

| 属性 | 内容 |
|---|---|
| 主なユーザー | 開発者自身（Naoya）および知人への限定公開 |
| 想定規模 | 少人数（数名〜十数名） |
| ターゲット像① | ビジネス現場で意思決定・問題解決に関わる社会人 |
| ターゲット像② | 自分の論理の飛躍や「伝わらなさ」に課題を感じている人 |
| ターゲット像③ | 書籍でロジカルシンキングを学んだが、実践で活かせていないと感じている人 |
| 現フェーズ | ビジネス向けに特化（受験モードは将来対応） |

**ターゲットから除外する像**：「正しい日本語・文章術を身につけたい人」は対象外とする。
このアプリの目的は「論理の構造を鍛えること」であり、文章の正確さ・美しさを目的にする人とはコンセプトが異なる。

## 3. 論理力3軸と「伝える力（B）」の位置づけ

要件定義では論理力3軸として「読む力（A）・判断する力（D）・伝える力（B）」を定義している。
ただし現在の論理トレーニング（logic.html）には「伝える力（B）」に対応するタブが存在しない。

### 3-1. なぜ「伝える力」タブがないのか

以下の段階を経て、意図的に論理トレーニングから除外した。

**段階1（気配りタブの設計）**：
「伝わりやすい表現を意識する（端的・箇条書きなど）」を鍛える「気配りタブ」を設計した。

**段階2（気配りタブの拡張検討）**：
「課題の意図を組んで、先回りして提案する（代替案を出すなど）」という要素を追加しようとした段階で、この訓練には一方向の表現ではなく**会話のラリー形式**が必要と判明した。この時点で「気配りタブ」は単純なロジックトレーニングよりも複雑で独立したアプリに近い性格を持つことが明確になった。

**段階3（思考トレーニング導入後）**：
思考トレーニング（thinking.html）の設計を進める中で、「伝達モード（伝える力の訓練）」はトレーニングの趣旨からずれる要素（ノイズ）と判断し、完全廃止を決定した。

### 3-2. 現在の棲み分け

| 力 | 対応するサービス |
|---|---|
| 読む力（A） | 論理トレーニング（穴埋め・要約・批判読み） |
| 判断する力（D） | 論理トレーニング（批判読み・空雨傘）＋ 思考トレーニング（6タイプ） |
| 伝える力（B） | 思考トレーニングの⑤「相手を動かす」思考タイプで対応 |

**方針**：「伝える力（B）」は思考トレーニングの⑤「相手を動かす」思考タイプで扱う。
論理トレーニング単体での専用タブは現フェーズでは設けない。

> 思考タイプ⑤「相手を動かす」の詳細は `requirements/thinking/overview.md §3-2`、
> 採点上の扱いは `requirements/thinking/scoring.md §3` を参照。

## 4. 両サービスの位置づけと関係

```
論理トレーニング（logic.html）
  └ 与えられた情報を正確に処理する力を鍛える（読む・整理する・構造を把握する）
  └ 1問5〜10分で完結するトレーニング
  └「思考トレーニングへ →」のリンクを設置

思考トレーニング（thinking.html）
  └ 処理した情報から判断を導き、問いを立てる力を鍛える
  └ 1セッション15〜60分の深いトレーニング
  └「論理トレーニングへ ←」のリンクを設置

共有するもの：
  └ ペルソナ設定（localStorage）
  └ 言語設定（localStorage）
  └ GASのスプレッドシート（別シートで管理）
  └ 共通JS（js/shared/）
```

論理トレーニングは思考トレーニングの前提として推奨されるが、強制はしない。

## 6. AI設計の共通方針

このセクションはCursorおよびClaudeへの指針として明示する。
問題生成・採点・フィードバックの設計はすべてこの方針に基づくこと。

### 6-1. ビジネス題材の原則

- 全問題の問題文・設問文はビジネス現場で実際に起こりうる状況を題材にする
- 評論文・哲学的文章・論文調の題材は**ビジネスモードでは使用しない**（受験モードで使用）
- 業界を指定した場合でも、**その業界の専門知識・専門用語・規制・法律を知らなくても読める内容**にする。業界は「文脈の色付け」のみに使用する

### 6-2. 設問文・フィードバックの言葉遣い

- 設問文・フィードバックはビジネス現場で自然に使われる言葉に限定する
- 論理学の専門用語（演繹・帰納・対偶・論証など）は設問文・フィードバック文には使わない
- 「何をすればよいか」が直感的にわかる表現にする

### 6-3. 温度設定（temperature）の原則

| 用途 | 設定値 | 理由 |
|---|---|---|
| 問題生成 | 0.9 | 多様な問題が生成されるよう創造性を確保 |
| 採点・フィードバック | 0.3 | 採点基準が一貫するよう低めに設定 |

※ 思考トレーニングの振り返り対話（役割D1〜D3）は例外として0.7を使用する（自然な対話のため）。詳細は `requirements/thinking/scoring.md` を参照。

### 6-4. AIの立場：「採点者」ではなく「上司」として振る舞う

AIの立場には2つの選択肢がある。

- **採点者**：回答に正解・不正解の判定を下し、模範解答を示す
- **上司**：答えを教えず、論理の弱点に対して問い返しを行い、本人に気づかせる

このサービスでは後者を採用する。
理由は、ビジネス現場で人が成長するのは「正解を教わったとき」ではなく「上司から問い返しを受けて自分で気づいたとき」だからである。
特にコンサル業界では、上司との対話によって思考が鍛えられる経験がスキル形成の中核を成している。

AIが採点者として振る舞うと、ユーザーは「正解を覚える」学習になりやすい。
上司として振る舞うことで、ユーザーは「自分の思考の癖に気づき修正する」学習へと自然に誘導される。

この方針は論理トレーニングの採点フィードバックにも、思考トレーニングの問い返しにも共通して適用される。

## 7. ペルソナ設計の方針

### 7-1. ペルソナとは何か

「あなたはどんな人ですか？」を定義する概念。業界×職種・勤続年数で構成される。
1度設定すれば全タブ・全問題に共通で適用される。

**業界（テーマ）との違い：**

| 概念 | 問い | 範囲 | 目的 | 設定UI |
|---|---|---|---|---|
| ペルソナ | 「あなたはどんな人ですか？」 | 全タブ横断・全アプリ共通 | 回答者の背景に沿った表現で題材を生成 | ヘッダーのプロフィールボタン |
| 業界（テーマ） | 「この問題はどんな業界の話か？」 | 問題ごと | この問題の題材の文脈の色付け | 問題作成エリア |

両者は独立して機能し、混在を避けるためUI上も別の場所に配置する。

### 7-2. 問題生成への反映方針

ペルソナが設定されている場合、問題生成プロンプトに以下を付加する。

```
回答者のバックグラウンド：[業界×職種]（勤続[N]年）
このバックグラウンドに沿った題材・事例・表現レベルで問題を生成すること。
ただし難易度・採点基準はバックグラウンドによって変えないこと。
```

### 7-3. 勤続年数によるスキルレベルの解釈

| 勤続年数 | 解釈 |
|---|---|
| 1年未満 | 基本的な業界用語は知っているが実務経験は浅い |
| 1〜3年 | 日常業務の流れは把握しているが専門的判断は経験不足 |
| 3〜5年 | 専門知識を持ち、自律的に業務を遂行できる |
| 5〜10年 | チームや案件をリードできる経験を持つ |
| 10年以上 | マネジメント・意思決定の経験が豊富 |

## 8. 非機能要件

| 項目 | 内容 |
|---|---|
| パフォーマンス | 問題生成：通常30秒以内（API依存） |
| セキュリティ | Claude APIキーはリポジトリにコミットしない。ローカルは `js/shared/01-config.local.js`、本番は GitHub Actions の `CLAUDE_API_KEY` Secret で注入 |

| データ保持 | Google Spreadsheetに蓄積。削除操作で消去 |
| アクセス制限 | なし（URLを知っている人が利用可能） |
