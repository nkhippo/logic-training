## 1. このサービスの目的

断片的で曖昧な状況の中から「何を問うべきか」を自分で定義し、
仮説を積み重ねながら判断・行動を導く力を鍛えるWebアプリ。

AIとの対話を通じて思考プロセスを繰り返し積み上げることで、
仕事の現場で使える判断力と、問いを立てる力を育てることを目的とする。

---

## 2. なぜこのサービスが必要か

### 2-1. 仕事における根本的な難しさ

仕事で本当に難しいのは、答えを出すことではなく、
**何を問うべきかを決めること**にある。

どれだけ論理的に正確な答えを出しても、解くべき問いが間違っていれば、
その答えに意味はない。コンサルタントであれば「問いの設定を誤る」ことの
コストを肌で知っているはずである。クライアントへの最終プレゼンで、
そもそも解くべき問いが違っていたと気づいたときの後戻りは大きい。

実際のビジネスでは、問いが明確に与えられる場面はほとんどない。

- 状況は断片的で、情報はいつも不完全
- ステークホルダーの利害は複雑に絡み合う
- 「何を解くべきか」という問い自体、誰も定義してくれない

### 2-2. 書籍でロジカルシンキングを学んでも実践で活かせない理由

**問いが最初から用意されている**  
書籍の問題は「この文章の論点は何か」という形式をとる。
問いが最初から与えられている問題を解き続けても、
自分で問いを立てる力は育たない。

**情報が整理された単発問題しかない**  
書籍はリアルタイムのフィードバックができないため、
情報が整理されていて正解が1つに定まる問題に集中する。
曖昧な情報を前に仮説を積み重ねていくプロセスは、書籍では訓練できない。

**自分の思考への個別フィードバックがない**  
「どこで論理が飛んだか」「何を見落としたか」をその場で指摘してもらえる環境が、書籍には存在しない。

### 2-3. AIとの対話形式が必要な理由

**問いが与えられない問題を作れる**  
AIは「断片的な状況だけを提示し、何を解くべきかをユーザーが定義する」
問題を生成できる。こうした問題は、書籍では構造的に作れない。

**ユーザーの回答に応じて問いを動的に生成できる**  
「一次判断→新情報の追加→判断の修正→最終判断」という
段階的な思考プロセスを訓練するには、リアルタイムで反応できる対話相手が必要である。

**見落としをその場で指摘し、その代償を示せる**  
AIはユーザーの回答を読んで、どの思考ステップが抜けているかを検出し、
「その判断、3ヶ月後に覆る可能性はないか」という形で
見落としの影響を具体的に示すことができる。

## 3. 思考力の定義と設計思想

### 3-1. このトレーニングが最終的に目指すこと

このトレーニングの最終的なゴールは、
**自分で状況を読み、「何を解くべきか」という問いを立てられるようになること**である。

### 3-2. なぜ6つの思考タイプが必要か

最終的なゴールは「問いを立てられるようになること」だが、
問いを立てる力は判断のプロセスを知らないと育たない。

実際の仕事で判断や問題解決を行うとき、
意識しているかどうかに関わらず、思考の操作は以下の6つに集約される。

| タイプ | 何をするか |
|---|---|
| ①状況を整理する | 今何が起きているかを、事実・不確かな情報・矛盾に分けて読む |
| ②原因を掘り下げる | なぜそうなったかを、表面の原因から根本まで掘り下げる |
| ③先を読む | このまま進むとどうなるか、何が起きうるかを見積もる |
| ④選択肢を比べる | 何が最善かを、自分なりの基準を持って判断する |
| ⑤相手を動かす | 誰がどう動くかを想定し、伝え方・順序を設計する |
| ⑥問いを疑う | そもそも今解こうとしている問いが正しいかを立ち止まって確認する |

**6タイプを採用した理由：**
- **設問の再現性**：どんな業界・場面でも同じ問いかけの型が使えるため、AIへの指示が安定し、採点基準もブレにくい
- **学習の段階性**：レベル1で個々のタイプを体感し、レベル3〜4で組み合わせ・選択を鍛えられる

**6タイプの連鎖構造：**

```
①状況を整理する（現状把握）
　↓
②原因を掘り下げる / ③先を読む / ⑤相手を動かす
（状況に応じて縦に深めるか横に広げるか）
　↓
④選択肢を比べる（判断）
　↓
⑥問いを疑う（問い自体の検証）
```

**設計の経緯：**  
最初は「実現可否・優先順位・合意形成・原因分析・リスク見積もり」という
5タイプで整理していたが、これらは「何を問うか（問いの核心）」の分類であり、
「思考の操作」の分類ではないことに気づいた。
問いの核心（10タイプ）と思考タイプ（6タイプ）を分離することで、
設問の汎用性と採点の安定性が大幅に向上した。

### 3-3. なぜ4つのレベルが必要か

6タイプを「知っている」だけでは、仕事で使えるようにはならない。
「補助輪を段階的に外していく」設計が必要だった。

| レベル | 何が与えられるか | 自分で決めること | 適用難易度 |
|---|---|---|---|
| レベル1 | 題材文＋設問A・B・C | 特になし | 難易度1〜5 |
| レベル2 | 題材文＋使うタイプと順序 | 各タイプで何を確認するか | 難易度3〜5 |
| レベル3 | 題材文＋6タイプの一覧 | どのタイプをどの順序で使うか | 難易度3〜5 |
| レベル4 | 断片的・曖昧な題材文のみ | 問いを立てることから始めるすべて | 難易度3〜5 |

**各レベルの設計意図：**

- **レベル1**：型の使い方を体感する。6タイプの存在と使い方を知る入口
- **レベル2**：型の中で「何を確認するか」を自分で問う。型の背後にある目的意識が問われる
- **レベル3**：型を選ぶ。同じ状況でも「どの型で解くか」を誤ると判断が狂う
- **レベル4**：問いを生む。断片的な状況から「何を解くべきか」を定義することから始まる

### 3-4. なぜ難易度とレベルの2軸が必要か

当初は「難易度」の軸のみで設計しており、難易度が上がるにつれて題材の複雑さと設問の難しさの両方を上げる案を検討していた。

**問題意識（思考トレーニング企画の中期段階で判明）**：

「難しい題材に設問A・B・Cで答える」という形式では、どこまでいっても「与えられた問いに答える」訓練にしかならない。このサービスの最終ゴールである「問いを自分で立てる」という訓練は、形式を変えないと実現できないことが明確になった。

また、難易度軸に「問いを自分で立てる」要素を組み込んでしまうと以下の問題が生じる：
- 難易度5になるにつれて「複雑な題材 × 自分で問いを立てる」という超高難易度になる
- 「低難易度で自分で問いを立てる経験」ができなくなる
- **「まず簡単な状況で問いを立てる練習をしたい」というユーザーが行き場を失う**

**2軸に分離した解決策**：

難易度（題材の複雑さ）とレベル（問いの自律性）を独立した2軸として設計する。

```
難易度1 × レベル4  → 「シンプルな状況で自分で問いを立てる」練習
難易度5 × レベル1  → 「複雑な題材を与えられた設問で解く」練習
難易度3 × レベル3  → 「中程度の複雑さで、タイプを自分で選ぶ」練習
```

この2軸設計により「まず型を覚える（レベル1）→ 型の使い方を自分で考える（レベル2・3）→ 問いを自分で立てる（レベル4）」という習熟ルートを、あらゆる難易度で経験できる。

**なぜ4段階か：**  
3段階では「型の中身を定義する（レベル2）」と「型を選ぶ（レベル3）」が混在してしまい、
5段階では粒度が細かすぎて実装・運用コストが見合わない。
4段階は「型を使える→中身を定義できる→選べる→問いを生める」という
学習の自然な段階に対応している。

### 3-5. 問いを立てる力は繰り返すだけでは身につかない

反復で身につくのは「型の実行精度」にとどまる。
「型の選択判断」や「問いの定義」は反復だけでは育たない。

理由：型の選択や問いの定義には「なぜその情報が必要なのか」という
目的意識が必要。目的意識とは「この型を省いたら何が起きるか」を
具体的に理解していること。書籍では型を省いても採点に影響しないため、
「なぜその型が必要か」の理解が生まれない。

**目的意識を育む2つのプロセス：**

1. **失敗を体験すること**  
   型を省いた状態で判断を下したとき、AIが上司の立場から問い返す。  
   「その選択肢、3ヶ月後に覆る可能性はないか」  
   答えを教えるのではなく、省いた型の意味を自分で気づかせる。

2. **省略の代償を最終フィードバックで明示すること**  
   最終フィードバックで省いた型の具体的な代償を言語化して示す。  
   「②原因を掘り下げるを省くと、直接原因への対処しか行われず、同じ問題が繰り返す構造が残る」  
   この具体的な影響の言語化が「なぜその型が必要か」の理解につながり、
   次の問題に取り組むときの目的意識として機能する。

## 4. 問題設計の全体構造

### 4-1. 問いの核心（10タイプ）

問いの核心は「どんなビジネス場面の問いを解くか」を定義する補助概念。
問題を作る際にユーザーが任意で選択でき（デフォルトはランダム）、
題材文の生成精度と採点基準の一貫性を高めるために使用する。

- **ユーザーには見せず、内部処理にのみ使う**
- 1つの問題につき1つだけ選択する（複数同時選択は許容しない）

| 問いの核心 | 問いの本質 | 主に使う思考タイプ |
|---|---|---|
| 実現可否型 | これはできるか？ | ④＋①③ |
| 優先順位型 | どれを先にやるか？ | ④＋① |
| 合意形成型 | どうすれば通るか？ | ⑤＋①④ |
| 原因分析＋ゴール定義型 | なぜ起きたか・何を目指すか？ | ②＋①④ |
| リスク見積もり型 | 何が起きうるか？ | ③＋①④ |
| 価値の定義型 | それは本当に良いことか？ | ⑥＋①② |
| 構造の発見型 | なぜ繰り返すのか？ | ②＋① |
| 視点の転換型 | 別の立場から見るとどう見える？ | ⑤＋⑥ |
| トレードオフの明示型 | 何を諦めるか？ | ④＋③⑥ |
| 前提の問い直し型 | 当たり前は本当に正しいか？ | ⑥＋① |

### 4-2. 題材文の形式

デフォルトは以下の混在形式を採用する。

```
数値・実績などの定量的な情報
＋
会議での発言・メール・ヒアリング内容などの定性的な情報
```

レベルが上がるほど情報は断片的・曖昧になる設計とする。
レベル4では「最近、チームの動きが悪い気がする」というほぼ何も与えられない状況から始まる問題を生成する。

### 4-3. 難易度とレベルの対応

| 難易度 | レベル選択 |
|---|---|
| 1・2 | レベル1固定（UIでレベル選択を非表示にする） |
| 3〜5 | レベル1〜4をユーザーが自由に選択可能 |

### 4-4. 回答の流れと点数ルール

各ステップで回答するたびに、AIが内部で100点満点のスコアをつける。
**スコアの数値はユーザーには表示しない**（内部処理にのみ使用）。

```
95点以上         ：次のステップへ進む（フィードバックなし）
80点以上95点未満 ：次のステップへ進む
                  （1点だけ指摘を受け、直すか次へ進むかを選べる）
80点未満・1回目  ：AIから問い返しを受ける
80点未満・2回目  ：そこで終了
                  （次のステップには進まず、ここまでのフィードバックを受け取る）
```

### 4-5. 最後の問い「つまり、どういうこと？」

難易度2以上のすべてのレベルで、最後に共通して入る問いかけ。

**目的：**  
問題を通じて自分が定義した「解くべき問い」に対して、
「なぜその結論に至ったか・次に何をすべきか」を端的に言語化できるかを確認する。

**形式：**  
AIがユーザーの問いをもとに質問文を動的に生成する。  
例：「この依頼を3週間以内に実現できるかどうか、なぜそう判断したか・次に何をすべきかを2文・100字以内でまとめてください」

**点数ルール（90点ルール）：**

```
90点以上          ：最終フィードバックへ
90点未満・1回目   ：1回だけ問い返しを受ける
90点未満・2回目   ：最終フィードバックへ（終了ではなく通常のフィードバック）
```

**難易度1で除外する理由：**  
難易度1は「型の使い方を体感する」入門段階であり、
ここまでの分析を圧縮して言語化する負荷は学習目的に対して重すぎる。

## 5. 機能要件

### 5-1. 問題を作るときの設定

```
必須：
　難易度（1〜5）
　レベル（1〜4）　※難易度1〜2はレベル1固定・UI非表示

任意：
　問いの核心（10タイプから選択・デフォルトはランダム）
　　→ ユーザーには表示しない
　　→ 題材文の生成と採点基準の精度向上に使用
　　→ 過去問に紐づけて保存する
　業界（テーマ）
　ペルソナ（設定済みの場合は自動で反映）
```

### 5-2. 共通機能

| 機能 | 内容 |
|---|---|
| 問題生成 | Claude APIで題材文・設問文をその都度生成（設問文も固定テンプレートを使わない） |
| 採点・フィードバック | 役割A〜D3を分けて処理する（詳細は `requirements/thinking/scoring.md`） |
| 過去問の保存 | GAS経由でGoogle Spreadsheetに保存 |
| 言語切替 | ヘッダーの「言語」ボタン→モーダルで日本語・英語を選択 |
| ペルソナ設定 | 論理トレーニングと共有（localStorage） |
| 業界選択 | 全問題で共通・任意 |
| 振り返り機能 | `ENABLE_REFLECTION` フラグで制御（詳細は `requirements/thinking/scoring.md §3`） |

### 5-3. 設問文の動的生成方針

設問文はAIがその都度生成する設計とする。固定テンプレートを使わない。

**理由：**
- 題材ごとに具体的な人物・数値・発言を組み込んだ設問のほうが、ユーザーが「何を答えればよいか」を直感的に把握できる
- 固定テンプレートでは題材との一貫性が崩れ、誤回答を誘発しやすい

### 5-4. 過去問の保存方針

```
保存するもの：
　└ 題材文・設問
　└ 問いの核心（ユーザーには見せない・内部保持）
　└ user_core（システムデフォルトまたは過去のユーザー定義）
　└ 難易度・レベル
　└ 業界・ペルソナのスナップショット

保存しないもの：
　└ ユーザーの回答
　└ 採点フィードバック
　└ 振り返りの内容
```

**保存しない理由：**  
過去問を保存する主な目的はAPI利用コストの削減である。
他のユーザーが採点結果を参照して答え合わせに使えないよう、
回答とフィードバックは保存しない設計とする。
