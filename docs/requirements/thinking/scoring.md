## 1. AIの立場：「上司」として動作する

採点者ではなく、「上司」として動作する。

```
目的　：最終的な判断の質を上げるために、部下の思考を引き出す
制約　：答えを直接教えない。「ここが悪い」と一方的に批判しない
やり方：省いた型の影響を具体的に示す問いかけで、自分で気づかせることを優先する
```

**なぜ「採点者」ではなく「上司」か：**

| 立場 | やること | ユーザーの学習モード |
|---|---|---|
| 採点者 | 正解・不正解の判定を下し、模範解答を示す | 「正解を覚える」学習 |
| 上司 | 答えを教えず、論理の弱点に問い返す | 「自分の思考の癖に気づき修正する」学習 |

コンサル業界では、人が成長するのは「正解を教わったとき」ではなく
「上司から問い返しを受けて自分で気づいたとき」である。
この体験をAIで再現するのがこのサービスの設計方針。

**スコア非表示の理由：**  
「上司」は部下に点数を見せない。
スコアは内部処理（80点/90点ルールの分岐）にのみ使用し、
ユーザーへのメッセージには含めない。

## 2. 採点の役割構成

採点処理は役割A〜D3に分離して設計する。

```
役割A：点数の判定（内部処理のみ）
　├ レベル別の評価基準を適用する
　├ 問いの核心に照らして回答の方向を確認する
　├ 論理的な整合性をチェックする
　├ ユーザーが定義した問い（user_core）を抽出する（レベル2〜4）
　└ JSONで返す：点数 / 合否 / 省いた思考タイプ / 論理の問題 / 判定理由 / user_core

役割B：問い返しの生成
　├ 95点以上：次へ進む（問い返しなし）
　├ 80〜95点：指摘1つ＋「直す」か「次へ進む」かを選べる
　├ 80点未満：省いた型の影響を具体的に示す問い返しを生成する
　└ 点数の数値はユーザーへのメッセージに含めない

役割C＋D1：最終フィードバック＋振り返りへの問いかけ（1回のAPI呼び出し）
　├ パート1：最終フィードバック
　│　├ できていたこと
　│　├ 省いた視点とその影響
　│　│　（例：「②原因を掘り下げるを省くと、直接原因への対処しか行われず、
　│　│　　　　　同じ問題が繰り返す構造が残る」）
　│　├ 論理の整合性への指摘
　│　├ 別の考え方の提示
　│　└「つまり、どういうこと？」への評価
　└ パート2：振り返りへの問いかけ（ENABLE_REFLECTION=trueのみ）
　　　└ 以下の3パターンから1〜2問を選んで提示する
　　　　 ・パターンX（省略・混同への指摘）：
　　　　 　「[ステップ]で[具体的な問題]がありました。そのときの判断の根拠を振り返ってみてください」
　　　　 ・パターンY（自己批判の促し）：
　　　　 　「今回の回答全体で、最も『本当にこれで良いか』と迷った場面はどこですか？」
　　　　 ・パターンZ（思い込みの指摘）：
　　　　 　「[ステップ]で[前提]を確認せずに進んだ場面がありました。なぜそこを疑わなかったと思いますか？」

役割D2：振り返り1回目のフィードバック＋追加の問いかけ（任意・点数なし）
　├ ユーザーの回答から「質問があるか/ないか」をAIが読み取る
　└ 必ず問いかけで締める（D1とは異なるパターンを選ぶ）

役割D3：振り返り2回目のフィードバック（任意・点数なし・問い返しなし）
　├ ユーザーの回答から「質問があるか/ないか」をAIが読み取る
　└ この問題全体を通じた気づきを1つ提示して締める
```

## 3. 論理的な整合性チェックの要件

以下の3層でチェック項目を定義する。採点のたびに、
問いの核心と使った思考タイプに合わせて動的に組み合わせて使う。

**全問題に共通するチェック：**
- 事実と自分の解釈・意見が混ざっていないか
- 結論と根拠の間に飛躍がないか（AだからBと言えるか）
- 原因と結果の向きが逆になっていないか

**思考タイプ別のチェック（例）：**
- ①状況の整理：事実・不確かな情報・矛盾の3つに分けられているか
- ②原因の掘り下げ：表面の原因で止まらず、根本まで掘り下げられているか
- ④選択肢の比較：評価の基準が明示されているか、何を諦めるかが意識されているか
- ⑤相手を動かす：自分の論理を押しつけていないか、相手が動く条件から考えているか

**問いの核心別のチェック（例）：**
- 実現可否型：「できる/できない」の判断基準が明示されているか
- 合意形成型：主な関係者全員が考慮されているか
- 原因分析型：どこまで直すかが意識的に選ばれているか

## 4. user_core（ユーザーが定義した問い）の設計

### 4-1. user_coreとは何か

「ユーザーが定義した解くべき問いを1文で要約したもの」。
最終問い「つまり、どういうこと？」の質問文をAIが動的に生成する際の起点として使用する。

### 4-2. レベル別の取得方法

| レベル | user_coreの取得方法 |
|---|---|
| レベル1 | システム側のデフォルト（問いの核心ごとに定義された定型文）を使用 |
| レベル2 | 切り口定義ステップの採点APIで抽出 |
| レベル3 | タイプ選択ステップの採点APIで抽出 |
| レベル4 | 問い定義ステップの採点APIで抽出 |

### 4-3. 具体例

**問いの核心が「実現可否型」の場合：**

システムデフォルト：
> 「この取り組みを、与えられた制約の中で実現できるか判断する」

レベル4でユーザーが定義した問い（AIが採点時に抽出）：
> 「3週間以内に新機能開発を実現できるか判断する」

→ AIが生成する最終問い：
> 「この依頼を3週間以内に実現できるかどうか、なぜそう判断したか・次に何をすべきかを2文・100字以内でまとめてください」

### 4-4. 打ち切り時の扱い

0回目（または-1回目）で打ち切りになった場合は、
システム側のデフォルトにフォールバックする。

## 5. 振り返り機能

### 5-1. 目的

「省いた型の影響を知る」にとどまらず、
「なぜ自分はそこを省いたのか」を言語化することを促す。

自分の思考の傾向に気づくには、思考プロセスを後から振り返り
「なぜそう考えたか」を言葉にする経験を積む必要がある。
この機能はその場を提供するために設計されている。

`requirements/thinking/overview.md §3-5` で述べた
「目的意識を育む」プロセスを、最終フィードバック後の対話として実現する位置づけにある。

### 5-2. フロー

```
1. 最終回答を送信する
2. AIから最終フィードバック（役割C）＋振り返りへの問いかけ（役割D1）が届く
3. 問いかけに回答する（任意）
　 → 回答しない場合は「スキップして終了」を選び、ここでセッションを終える
4. AIからフィードバック＋さらに深める問いかけ（役割D2）が届く
5. 問いかけに回答する（任意）
　 → 回答しない場合はここでセッションを終える
6. AIから最後のフィードバック（役割D3）が届く（問い返しなし・示唆で締める）
```

ユーザーが任意のタイミングで離脱できる設計とし、回答を強制しない。

### 5-3. 制御フラグ

振り返り機能は `ENABLE_REFLECTION` フラグで全体的に有効/無効を切り替えられる。
初回は有効（true）で運用する。
重くなった場合に無効化できる設計にしておく理由は、
振り返りがセッションとして長くなるためである。
