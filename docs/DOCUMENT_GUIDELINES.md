# thinkgrindai — ドキュメント記載粒度ガイドライン

**最終更新**: 2026-05-25  
**目的**: Claude・Cursorが要件定義書・仕様書を作成・修正するときに、本プロジェクトの記載粒度から逸脱しないようにする。

---

## 重要：このガイドラインの位置づけ

このガイドラインは、以下の2タイミングで**必ず**参照されること。

1. **Claudeが Cursor向けの作業指示書を作成するとき**：指示書内でドキュメント編集を含む場合、指示書冒頭でこのガイドラインへのリンクを必ず提示する
2. **Cursorが ドキュメントを編集するとき（直接指示も含む）**：作業前に必ずこのガイドラインを読み、`docs/TERMS.md` も併せて確認する

両者で二重にチェックされることで、ドキュメントの一貫性を保つ。

## 1. 基本原則

| 書類 | 問い | 記述範囲 | 読者 |
|---|---|---|---|
| **要件定義書**（`docs/requirements/`） | なぜ・何を | ユーザー体験・設計の背景・動機・目指す方向性 | Naoya・Claude（設計議論） |
| **仕様書**（`docs/specification/`） | どのように動くか | 実装・状態遷移・API仕様・データ構造・実装値 | Cursor（実装担当）・Claude（設計レビュー） |

---

## 2. 要件定義書に書くべきこと・書かないこと

### ✅ 書く

| カテゴリ | 内容 | 具体例 |
|---|---|---|
| **背景・課題感** | なぜこの機能が必要か | 「書籍では○○が解決できない」 |
| **ユーザー体験** | ユーザーが何をどう体験するか | 「テーマ・難易度を選んで問題を生成する」 |
| **学習目的** | このタブ・機能で何を鍛えるか | 「接続詞を文脈から選ぶことで論理的接続を読み取る」 |
| **設計の意思決定** | 複数の選択肢から選んだ理由 | 「対偶を除外した理由：ビジネスで使われないため」 |
| **概念の定義** | 用語・概念の定義 | 「user_coreとは何か」 |
| **段階性・学習体験** | 難易度・レベルで何が変わるか（ユーザー視点） | 「難易度1では接続詞のヒントが与えられる」 |
| **将来計画** | 今後のロードマップ | 「Phase 6で受験モードを追加する」 |

### ❌ 書かない

| 内容 | 正しい記載先 |
|---|---|
| 具体的なAPI呼び出し方法（max_tokens値・temperatureの数値） | 仕様書 |
| JSONの構造 | 仕様書 |
| 状態遷移フロー図 | 仕様書 |
| GASのカラム定義 | 仕様書 |
| コードの変数名・関数名 | ソースコード内コメント |
| 採点プロンプトの具体的な文言 | 仕様書（設計意図のみ） |

### 🔄 グレーゾーンの扱い方

| 内容 | 要件定義書での扱い | 仕様書での扱い |
|---|---|---|
| **難易度別の数値表（文字数・設問数）** | 学習体験の段階感を記述（数値表ではなく） | 実装値の数値表 |
| **設問タイプ一覧** | どんなタイプがあるか・なぜ必要か | 出題難易度・実装条件 |
| **採点基準** | 採点の考え方・思想 | 分岐条件・パラメータ |
| **エラーハンドリング** | 思想（ユーザー体験への影響） | 具体的な挙動 |

**判断軸**：「ユーザーが知るべき情報か、実装者が知るべき情報か」で分ける。
両者にまたがる場合は、要件定義書で「段階感」を、仕様書で「数値表」を書き、両者を相互参照する。

## 3. 仕様書に書くべきこと・書かないこと

### ✅ 書く

| セクション | 内容 |
|---|---|
| **状態遷移フロー** | 初期状態 → API呼び出し → 画面表示 → ユーザー操作 → ...の図解またはテキスト |
| **データ構造** | JSON形式・GASカラム・状態オブジェクトの構造 |
| **API仕様** | 関数名・パラメータ・戻り値・temperature・max_tokens の具体値 |
| **プロンプト設計意図** | temperature・max_tokens を選んだ理由（具体プロンプトは書かない） |
| **UI要素の挙動** | ボタンの配置・無効化条件・バリデーション・エラーメッセージ |
| **分岐条件** | 「score >= 95 ならば」のような条件分岐 |
| **依存関係** | 「`js/shared/04-industry-persona.js` に依存」のようなコード依存 |

### ❌ 書かない

| 内容 | 正しい記載先 |
|---|---|
| 「なぜこの機能が必要か」の背景 | 要件定義書 |
| ユーザー学習体験の哲学的な説明 | 要件定義書 |
| 具体的なプロンプト文言（全文） | ソースコード（`buildPrompt()` 等） |
| 設計判断の経緯（没案など） | `DESIGN_DECISION_HISTORY.md` |

## 4. ドキュメントの粒度判定チェックリスト

ドキュメントを編集・追加するときは、以下を確認すること。

### 4-1. 編集前のチェック

- [ ] `docs/TERMS.md` で関連用語の定義を確認したか
- [ ] 編集対象が要件定義書か仕様書か明確か（迷ったらこのファイルの §2・§3 を再読）
- [ ] 既存の類似セクションと整合性が取れるか

### 4-2. 編集中のチェック

- [ ] 要件定義書に「具体的な数値」を書こうとしていないか → 仕様書に置く
- [ ] 仕様書に「ユーザー体験の哲学」を書こうとしていないか → 要件定義書に置く
- [ ] グレーゾーンの内容は、相互参照で対応しているか
- [ ] 設計判断の経緯は `DESIGN_DECISION_HISTORY.md` に追記したか

### 4-3. 編集後のチェック

- [ ] セクション番号の連番が崩れていないか（前後のセクションも確認）
- [ ] 他ファイルからの crossref が壊れていないか（`grep -rn "§"` で検索）
- [ ] `docs/ai-context/FILE_MAP.md` の更新が必要か判断したか
- [ ] 関連する `cursor-instructions/` の指示書も更新が必要か判断したか

## 5. crossref（相互参照）のルール

ドキュメント間でセクションを参照するときの記法。

### 5-1. 正しい記法

```markdown
詳細は `requirements/common.md §3-1` を参照
詳細は `specification/thinking/api.md §2` を参照
```

- ファイルパスは `docs/` からの相対パス
- バッククォートで囲む
- セクション番号は `§N-M` 形式

### 5-2. crossref修正時の必須作業

セクション番号を変更した場合、**必ず**他ファイルでの参照を検索・修正すること。

```bash
# 検索コマンド
grep -rn "common\.md §" docs/
grep -rn "overview\.md §" docs/
```

セクションを追加・削除した場合は、後続セクション番号もすべて連動修正すること。

## 6. 「正」を一箇所にする原則

同じ内容を複数箇所に書く場合、必ず「正」を1箇所に定め、他は参照とする。

### 例

❌ 悪い例（同じ内容を2箇所に書く）

```markdown
# specification/thinking/overview.md
extraInfoは「一次回答後に新しい情報を提示する...」（詳細な説明）

# specification/thinking/data.md
extraInfoは「一次回答後に新しい情報を提示する...」（重複した説明）
```

✅ 良い例（参照に統一）

```markdown
# specification/thinking/overview.md（正）
extraInfoは「一次回答後に新しい情報を提示する...」（詳細な説明）

# specification/thinking/data.md（参照）
extraInfoの詳細は `specification/thinking/overview.md §6` を参照。
```

## 7. ドキュメント追加・削除時のチェックリスト

### 7-1. 新規ドキュメント追加時

- [ ] ファイル先頭に「最終更新」「バージョン」「目的」を記載
- [ ] `docs/ai-context/FILE_MAP.md` のドキュメント一覧に追加
- [ ] `docs/TERMS.md §1` のテーブルに追加（要件定義書・仕様書なら必須）
- [ ] 既存ファイルでこのドキュメントを参照すべき箇所がないか確認

### 7-2. ドキュメント削除時

- [ ] このドキュメントを参照している他ファイルがないか `grep` で確認
- [ ] 参照元の修正を同時に行う
- [ ] `docs/ai-context/FILE_MAP.md` から削除
- [ ] `docs/TERMS.md` から削除（記載があれば）

### 7-3. ドキュメント分割・統合時

- [ ] 旧ファイルにdeprecation noticeを追加（参照リンク付き）
- [ ] `docs/ai-context/FILE_MAP.md` を更新
- [ ] `docs/TERMS.md §1` を更新

## 8. ガイドライン違反時の動作

このガイドラインに違反する記載を見つけた場合、**修正前に必ずNaoyaに報告**すること。
特に以下のケース：

- 要件定義書に「実装値」が書かれている
- 仕様書に「学習体験の哲学」が書かれている
- 同じ内容が複数箇所に重複している
- crossrefが壊れている（セクション番号が存在しない）

軽微な修正（typo修正等）は除く。
