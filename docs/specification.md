### 2-3. 新規生成画面のテーマ選択UI（全タブ共通）

各タブの「新規生成」カード内で、上から次の順に並べる。

1. **テーマ**（必須）— タブごとのプリセット（穴埋め・要約・批判読み＝文書タイプ、空雨傘＝テーマ領域、気配り＝場面タイプ）。UIラベルは全タブ「テーマ」で統一。状態は `fDocType` / `sDocType` / `cDocType` / `aDocType` / `kibariScene` に保持。初期は未選択。難易度未選択時は全プリセットが選択可能。難易度選択後は `minDiff` に応じて無効化（**5列レイアウト**・右端余白なし）。
2. **業界**（任意）— 全タブ共通の `st.industry`（気配り含む）。初期は未選択。生成時に `industry` として問題に保存し、問題セクションのメタ行に表示（未選択時は「未選択」）。
3. **難易度**（必須）— 5段階。初期は未選択。説明欄は問題方針のみ。
4. **ボリューム**（要約タブのみ）— 難易度4以上のとき、難易度の下に表示。

**問題生成成功後**：`resetGenConditions()` でテーマ・業界・難易度・ボリューム等を初期化。

**生成ボタン**：テーマ・難易度未選択時は押下で `alert`（API非呼び出し）。

業界は文脈の色付けのみ。全タブの生成プロンプトに業界制約を付与。

### 2-6. 写真による回答・過去問（pp）スコープ

| タブ | 新規生成 | 過去問（pp） | 理由 |
|---|---|---|---|
| 穴埋め | テキストのみ | テキストのみ | 情報量が写真添削に足りない |
| 要約 | テキスト or 写真 | テキスト or 写真 | 長文回答の添削想定 |
| 批判読み | テキスト or 写真 | テキスト or 写真 | 同上 |
| 空雨傘 | テキスト or 写真 | テキスト or 写真 | 同上 |
| 気配り | テキスト（Markdown 等） | テキスト（`kbp-*` UI） | リッチ仕様のため要約系 pp と共通化しない |

過去問 UI（`pp-*`）は要約・批判読み・空雨傘で `st.answerMode` / 写真配列を共有する。**意図的**（同一タブ横断の回答方式）。

### 3-5. 気配りタブ（kibari）

#### 問題の流れ

1. 状況提示（AI生成）→ ユーザーがメッセージを書く
2. AIが読み手として反応
3. ユーザーが書き直す（難易度に応じて最大3往復）
4. 最終採点・フィードバック

#### 状況文と回答者への作業指示（5W1H）

| 要素 | 役割 |
|---|---|
| `situation` | ビジネス場面の説明。**When／Who／What／Where** が文中または文脈から読み取れること（明示的でなくてもよい） |
| `writeInstruction` | 初回回答欄の見出しに表示する**回答者向けの具体的な作業指示**（例：「本日中に、〇〇と△△に、この情報を社内チャットで共有するメッセージを記載してください。」） |
| `rewriteInstruction` | 2回目以降の見出し（読み手反応後の書き直し用）。同じ When/Who/What/Where を維持しつつ、不足点の補完を促す |
| `openingPhrase` | 冒頭の定型文（参考表示・入力不要・文字数対象外）。読み手・状況に合わせて問題ごとに変える |
| `closingPhrase` | 結びの定型文（同上） |
| `constraint` | **本文**の文字数制限など（作業指示の補足として併記可。冒頭・結びは含めない） |

- **Why** は状況文から読み手が判断できるため、作業指示では省略してよい
- 場面タイプに応じた動詞：報告・共有→共有／報告、依頼・指示→依頼／指示、提案・説明→提案／説明、自己表現→自己紹介／伝える
- **難易度基準への影響**：なし（往復回数・読み手数・利害関係・観点数・文字数制限は従来どおり）。作業指示の明示度のみ難易度に連動（1〜2は5W1Hをより明示、5はやや簡潔でも実行可能な指示にする）

#### 難易度と往復回数

| 難易度 | 最大往復回数 | 読み手 |
|---|---:|---|
| 1 | 1回 | 1人・シンプル |
| 2 | 1回 | 1人・軽微な利害関係あり |
| 3 | 2回 | 2人・立場が異なる |
| 4 | 2回 | 2〜3人・対立する利害関係あり |
| 5 | 3回 | 複数・感情への配慮が必要 |

#### 場面タイプ（プリセット）

| キー | 日本語 |
|---|---|
| report | 報告・共有 |
| request | 依頼・指示 |
| proposal | 提案・説明 |
| self | 自己表現 |

#### GASシート：kibari

| カラム | 内容 |
|---|---|
| id | タイムスタンプ |
| theme | テーマ |
| diff | 難易度 |
| scene | 場面タイプ |
| date | 保存日時 |
| industry | 業界（任意・未選択時は空） |
| situation | 状況説明 |
| readers | 読み手（JSON配列） |
| points | 盛り込むべき観点（JSON配列） |
| constraint | 文字数制限など |
| writeInstruction | 初回の回答者向け作業指示（5W1H） |
| rewriteInstruction | 書き直し時の作業指示 |
| openingPhrase | 冒頭定型文（参考表示） |
| closingPhrase | 結び定型文（参考表示） |
| firstAnswer | 初稿（冒頭＋本文＋結びの全文） |
| feedback | 採点フィードバック |
| lang | 言語 |

> 過去問保存は生成時に状況・作業指示を保存し、採点完了時に `firstAnswer` と `feedback` を更新する。

#### 過去問の演習UI

- 新規生成と同じ構成：状況 → 作業指示付き回答欄 → 読み手反応（難易度に応じた往復）→ 採点
- `writeInstruction` / `rewriteInstruction` が空の旧レコードは、場面タイプ・読み手・状況文から導出したフォールバック文言を見出しに表示（「メッセージを書いてください」のみの表示はしない）
- DOM ID は `kbp-*`（過去問）と `kb-*`（新規生成）で分離し、状態は `st.kibariPast` に保持

### 5-1. 採点時の `max_tokens`（答え合わせ）

採点レスポンスが途中で切れないよう、全タブ共通で従来上限の **約1.5倍** を `app.js` の定数で管理する。問題生成の `max_tokens` は対象外。

| 用途 | 条件 | max_tokens（従来 → 現在） |
|---|---|---|
| 穴埋め・批判読み・空雨傘 | 難易度1〜3 | 1500 → **2250** |
| 穴埋め・批判読み・空雨傘 | 難易度4〜5 | 2500 → **3750** |
| 要約（テキスト・写真） | 文章量 ≤500字 | 1200 → **1800** |
| 要約 | 文章量 ≤2000字 | 3000 → **4500** |
| 要約 | 文章量 >2000字 | 6000 → **9000** |
| 写真採点（要約以外のデフォルト） | — | 2500 → **3750** |

実装：`GRADE_MAX_TOKENS` 定数と `gradeMaxTokensByDiff()` / `gradeMaxTokensBySummaryLength()` を `callClaude` / `callClaudeMsg` の第3引数に渡す。
