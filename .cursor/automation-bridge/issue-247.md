### 具体的な修正内容

「### Step 3-a: ドキュメント整備 ＋ 点検」内「点検カテゴリ B」の「影響度の基準」を以下のように更新：

```markdown
影響度の基準：

- **大**：ユーザーの体験フローが変わる / 既存の運用手順が変わる
  - 例: 画面遷移の順序変更、自動表示→ボタン押下への変更、必須項目の追加、開発フローのステップ追加・削除
- **中**：見た目・文言が変わる / 任意の操作手順が変わる
  - 例: ボタンラベル・エラーメッセージのリライト（同じ意味を保つ範囲）、配色・余白・サイズの変更、任意項目の追加
- **小**：内部実装のみの変化（ユーザーには見えない）
  - 例: リファクタリング、変数名変更、内部関数の分割、ログ出力の追加

### 「大」と「中」の境界判断ルール

以下の **いずれか**に該当すれば**大**に分類する（Cursor の主観で「中」に下げてはならない）：

1. **意味が変わる文言修正**：文章の意味・トーン・指示内容が変わる（例：「採点が厳しいです」→「採点を緩めました」）
2. **操作完了までのステップ数増減**：ユーザーが目的を達成するまでのクリック数・入力数が変わる
3. **エラー時の表示有無**：エラーメッセージが新規表示される / 既存表示が消える
4. **既存ユーザーの再学習が必要**：これまでの使い方が通じない、再度マニュアル参照が必要になる

判断に迷う場合は **`Naoya/Claude が判断する事項`** として持ち帰る（カテゴリ B 影響度「大」相当扱い）。
```

### 補足：「迷ったら大」原則の明示

「影響度の基準」表の直下に以下の注記を追加：

```markdown
> ⚠️ **影響度の判断に迷ったら「大」とする**（持ち帰り側に倒す）。
> Cursor は持ち帰り判定の閾値を自己判断で下げない。
> 過剰持ち帰りは Naoya/Claude が判断時に「中」に再分類できるが、誤って実装が進んだ後の差し戻しコストは大きい。
```
